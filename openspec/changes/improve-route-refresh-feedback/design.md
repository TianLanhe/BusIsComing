## Context

主頁結果列表目前由 `SwipeRefreshLayout` 包裹 `RecyclerView`。用戶在列表頂部下拉後，`MainActivity.refreshCurrentResults()` 復用最近一次常用路線或臨時查詢上下文，並透過現有 `querySequence`、`isQueryInProgress` 和 `RouteResultsRefreshPolicy` 防止過期結果及重複查詢。

現有 `showRefreshLoadingState()` 只設定 `SwipeRefreshLayout.isRefreshing = true`。原生刷新圈與下拉內容綁定，刷新期間上下滾動時可能離開視線；`onInitialRoutes()` 返回後則立即結束 loading，沒有成功確認。路線基礎結果之後仍會透過獨立回調漸進補齊 ETA 和站點預覽。

路線詳情目前在 `BottomSheetDialog` 內使用一般 `ScrollView`。當大量途經站展開後，一般 `ScrollView` 無法與 `BottomSheetBehavior` 建立正確的巢狀滾動協議；內容仍未回到頂部時，向下手勢可能先被 BottomSheet 攔截並收合整個彈層。

新增、編輯、複製路線及臨時查詢共用 `PlaceInputController`，但候選結果透過 `MaterialAutoCompleteTextView.showDropDown()` 顯示為獨立 `PopupWindow`。即使 Activity 已採用 `adjustResize`，候選視窗仍可能與輸入法分層重疊；候選較多時，底部項目會被輸入法覆蓋。問題不應以固定鍵盤高度或單純調整 windowSoftInputMode 解決。

本 change 遵循 `docs/ui-style-guide.md` 的「安靜實用的現代通勤工具」風格，保持 XML + AppCompat + Material Components 技術棧，並以最小改動沿用既有查詢流程。

## Goals / Non-Goals

**Goals:**

- 讓刷新正式觸發後的進行中狀態在列表滾動期間始終可見。
- 明確區分刷新進行中、成功和失敗，同時避免阻塞列表閱讀與操作。
- 保留既有刷新語義：復用查詢上下文、保留排序、不記錄路線使用次數並防止並行查詢。
- 成功後展示最新結果並回到列表頂部；失敗時完整保留刷新前畫面。
- 讓長路線詳情可可靠地向下及向上滾動，內容未到頂時不被 BottomSheet 下拉收合手勢搶占。
- 讓起終點候選在輸入法顯示時仍有足夠可視空間，且可直接滾動和點選。
- 統一新增、編輯、複製路線及臨時查詢的候選列表、聚焦、返回與空白點擊行為。
- 以模擬器完成下拉手勢、浮層、滾動、成功／失敗和空結果的必要驗收。

**Non-Goals:**

- 不改動首次點擊 `查詢` 的整頁 loading 狀態。
- 不改動 Citybus API、解析器、repository callback 協議、ETA／站點預覽資料來源或補全順序。
- 不新增自動刷新、背景輪詢、骨架屏、結果卡片重設計或第三方動畫依賴。
- 不因主列表刷新主動關閉已開啟的路線詳情或 ETA 彈層，也不把彈層切換到另一條路線。
- 不改動路線詳情的資料來源、解析、時間線、折疊、重試或快取語義。
- 不改動地點搜尋 API、debounce、最多 100 條候選、loading／無結果／失敗文案、選擇校驗、交換、保存或查詢語義。
- 不重新設計整個路線編輯頁或臨時查詢彈層，也不引入 Compose 或第三方候選元件。

## Decisions

### Decision: 手勢階段使用原生指示器，查詢階段切換固定浮層

用戶下拉但尚未正式觸發刷新時，保留 `SwipeRefreshLayout` 原生、跟隨手勢的刷新圈，維持熟悉的刷新門檻反饋。`onRefresh` 觸發後隱藏原生持續刷新狀態，改由結果列表容器頂部中央的自訂浮層顯示不定進度。

結果列表區使用可疊放子 View 的容器承載 `SwipeRefreshLayout` 和浮層。浮層不放入 RecyclerView item，避免因 ViewHolder 回收、列表位置或滾動而離開視線；浮層本身不可點擊且不攔截觸控。

替代方案：

- 只使用原生刷新圈：改動較少，但無法保證滾動期間持續可見。
- 在列表上方增加固定刷新列：不遮擋內容，但不符合已確認的「浮在首張可見卡片上」視覺方向。

### Decision: 浮層顯示時為列表預留約 44dp 頂部空間

刷新進行中與成功確認期間，結果列表增加約 44dp 頂部 padding，讓首張可見卡片下移，避免頂部中央浮層遮住路線或候車資訊。浮層結束後恢復原 padding。

調整 padding 時應保留既有左右與底部 padding，且不改結果卡片自身尺寸。失敗收尾需保留刷新前的可見位置與偏移，不因移除浮層跳到列表頂部。

替代方案：

- 不預留空間並直接覆蓋卡片：視覺更緊湊，但可能遮擋首卡核心資訊。

### Decision: 在 MainActivity 內管理最小刷新視覺狀態

刷新視覺採用 `Idle`、`Refreshing`、`Success` 三個明確狀態：

- `Idle`：浮層隱藏，列表使用原 padding，可按既有條件下拉刷新。
- `Refreshing`：顯示固定進度浮層，保留舊結果，停用查詢按鈕與再次下拉刷新。
- `Success`：浮層切換為成功勾號，維持查詢與刷新停用 500ms，之後回到 `Idle`。

狀態管理優先放在 `MainActivity` 或同層輕量 helper，沿用現有 Activity + formatter／policy 風格；不為此引入 ViewModel、Compose 或新的架構層。Activity 銷毀、切換路線、清空結果或新查詢使 generation 失效時，必須取消尚未執行的 500ms 收尾。

成功圖示使用專案內向量 drawable 或既有 Material 能力，不新增第三方 UI 依賴。浮層需提供可被無障礙服務理解的「正在刷新」與「刷新完成」狀態描述。

### Decision: 基礎路線返回即完成刷新

沿用 `BusRouteQueryCallback.onInitialRoutes()` 作為刷新成功點。ETA 與站點預覽仍在背景漸進補齊，不延長固定刷新圈，也不修改 repository callback。

原因：

- 下拉刷新主要確認路線重新查詢已被執行並返回。
- 若等待所有 ETA／預覽，需擴充完成協議，且個別補全失敗可能讓刷新狀態長時間不結束，不符合最小改動。

### Decision: 成功立即採用最新結果，再顯示 500ms 成功確認

基礎路線返回非空結果時：

1. 立即使用刷新時最新的排序字段與方向替換列表。
2. 更新 `更新時間：HH:MM:SS`。
3. 將列表滾動到 position 0。
4. 將浮層由進度切換為成功勾號。
5. 500ms 後隱藏浮層、恢復列表 padding、查詢按鈕和下拉刷新。

基礎路線返回空結果時，先在舊結果上顯示成功勾號 500ms，再切換到既有「暫無可用巴士路線」狀態。這保持成功回饋一致，同時不把空結果誤判為請求失敗。

成功確認期間不接受另一個查詢或下拉刷新，避免延遲收尾修改新狀態。已開啟的詳情或 ETA 彈層保持原先所選路線，不因主列表重排或回到頂部而切換。

### Decision: 失敗完整保留刷新前畫面

刷新請求失敗時立即回到 `Idle`，顯示既有「刷新失敗，請稍後重試」Toast，並保留：

- 原路線結果與排序狀態。
- 原 `更新時間`。
- 原滾動位置與偏移。
- 已開啟彈層和當前查詢上下文。

失敗不展示成功勾號，也不清空列表或切換整頁失敗狀態。

### Decision: 刷新期間保留非衝突操作

浮層不攔截列表觸控。用戶可以繼續上下滾動、調整排序、打開路線詳情或 ETA 彈層；刷新完成後使用當時最新的排序字段與方向展示新結果。

沿用現有 `setQueryLoading(true)` 行為，`查詢` 按鈕停用並顯示 `查詢中...`。再次下拉刷新或點擊查詢不得發起第二個並行請求。

### Decision: 路線詳情使用支援 BottomSheet 的巢狀滾動容器

將詳情內容的一般 `ScrollView` 替換為 AndroidX `NestedScrollView` 或現有技術棧中的等效巢狀滾動容器，讓它成為 `BottomSheetBehavior` 可識別的滾動子 View。詳細內容展開後：

- 內容未到頂時，向下手勢只用於將詳情內容滾回頂部，BottomSheet 不應先行收合。
- 內容已到頂時，繼續向下拖動可沿用現有 BottomSheet 收合或關閉行為。
- 向上手勢先滾動詳情內容；彈層仍保留既有初始高度、長內容展開、拖動關閉和右上角關閉按鈕。

不改動載入、失敗、重試、途經站折疊、時間線、API、解析器或快取。

### Decision: 候選結果改為由 PlaceInputController 管理的表單內嵌列表

停止使用 `MaterialAutoCompleteTextView` 的獨立下拉視窗顯示成功候選結果，改由共用 `PlaceInputController` 驅動輸入框下方的內嵌 `RecyclerView`。列表仍使用單行地點名稱並最多展示接口返回的 100 條候選。

同一時間只顯示目前聚焦輸入框的候選列表。切換起點和終點焦點時，舊列表關閉，新列表在對應輸入框下方展示。搜尋 loading、無結果及失敗仍沿用目前輸入區域的位置和文案；只有「有候選結果」的容器由 PopupWindow 改為內嵌列表。

`RouteEditActivity` 內不再使用且與共用控制器重複的舊搜尋私有實作可一併移除，避免兩套候選邏輯繼續漂移；這屬於本次改動直接涉及的最小清理，不延伸重構其他頁面。

### Decision: 依輸入法可視區限制候選高度

透過 `WindowInsetsCompat.Type.ime()` 取得輸入法遮擋範圍，根據輸入法上方實際可視高度計算候選列表上限，不假設固定鍵盤高度。候選列表最大高度約為可視區的 40%，並保證至少 3 個完整候選項目可見及可點選；候選較多時由列表自身滾動。

如果目前表單位置不足以同時顯示聚焦輸入框和至少 3 個候選項目，外層表單需自動滾動將兩者帶入可視區。路線新增、編輯和複製共用此行為；候選關閉後不強制恢復先前外層滾動位置。

### Decision: 臨時查詢與一般表單採用各自既有容器調整

臨時查詢底部彈層顯示內嵌候選列表時，自動展開至接近全螢幕，以提供穩定的輸入法上方空間；候選關閉後恢復由內容決定的既有自然高度。新增、編輯及複製路線則由可滾動表單將目前輸入框和候選列表帶入可視區，不改 Activity 導航與保存流程。

### Decision: 選擇、空白點擊與返回優先處理候選列表

用戶選擇一端地點後：

- 如果另一端尚未選擇，保留輸入法並自動聚焦另一端輸入框。
- 如果起點和終點都已選擇，不自動切換焦點、提交查詢或收起輸入法。

候選列表顯示時，點擊表單空白處或第一次按系統返回，只關閉候選列表並保留目前輸入文字；只有候選列表已關閉時，返回才沿用頁面退出或 BottomSheet 關閉行為。此優先級由共用控制器或頁面薄層統一協調，避免新增／編輯／複製與臨時查詢出現差異。

## Risks / Trade-offs

- [Risk] 動態增加與恢復頂部 padding 可能造成列表位置跳動。→ Mitigation: 保留原 padding，失敗路徑記錄並恢復可見 item 與 offset；成功路徑按需求明確回到 position 0。
- [Risk] 500ms 延遲收尾可能在切換路線或 Activity 結束後執行。→ Mitigation: 將收尾 Runnable 綁定目前 query generation，並在清空、失效及生命週期結束時取消。
- [Risk] 刷新期間排序舊結果，完成時可能與刷新開始時排序不同。→ Mitigation: 提交新結果時讀取當下最新排序字段與方向，而不是固定使用刷新開始快照。
- [Risk] 空結果延遲 500ms 才展示空狀態。→ Mitigation: 延遲固定且短，只用於一致的成功確認；期間保持輸入停用以避免狀態競爭。
- [Risk] 浮層在不同字體與螢幕尺寸下可能遮擋內容。→ Mitigation: 保持緊湊固定尺寸、預留約 44dp 頂部空間，並把模擬器多尺寸／字體視覺驗證列為完成門檻。
- [Risk] `NestedScrollView` 與 BottomSheet 的手勢邊界仍可能受內容高度和展開狀態影響。→ Mitigation: 以長路線、多次展開及內容非頂部向下滑動建立 instrumentation 回歸，並在模擬器驗證內容到頂前彈層不收合。
- [Risk] 內嵌 `RecyclerView`、外層表單滾動與輸入法 inset 同時變化可能造成跳動。→ Mitigation: 候選高度集中計算，只在焦點、候選資料或 IME inset 變化後重新佈局，並避免關閉列表時強制恢復外層位置。
- [Risk] 自動聚焦另一端可能形成重複搜尋或焦點迴圈。→ Mitigation: 只有另一端尚未選擇時執行一次焦點轉移，並沿用 controller 的 generation／debounce 機制忽略過期結果。
- [Risk] 返回鍵處理可能與 Activity 或 BottomSheet 關閉競爭。→ Mitigation: 先由候選可見狀態消費第一次返回；列表隱藏後再交還既有返回分派。
