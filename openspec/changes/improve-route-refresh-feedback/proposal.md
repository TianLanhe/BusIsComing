## Why

目前多個路線查詢介面存在操作已發生但反饋不穩定的問題：下拉刷新指示器可能隨結果列表滾動離開視線；路線詳情展開大量途經站後，向下手勢會先拖動整個底部彈層，導致內容難以返回頂部；起終點候選地點使用獨立下拉視窗，可能被輸入法遮擋而無法直接查看和點選。需要在保留現有資料流程與主要互動的前提下，修正這三類 UI 反饋和手勢優先級問題。

## What Changes

- 保留下拉手勢階段的原生刷新圈；正式觸發後切換為固定在結果列表頂部中央的浮層刷新指示器。
- 刷新期間保留舊結果並允許列表滾動、排序與卡片操作，同時阻止重複刷新和另一個查詢。
- 固定浮層顯示期間為列表增加頂部空間，避免遮擋首張可見路線卡。
- 以基礎路線結果返回作為刷新完成點，不等待後續 ETA 與站點預覽漸進更新。
- 刷新成功後立即替換結果、更新時間並回到列表頂部；浮層切換為成功勾號 500ms 後消失。
- 刷新成功但沒有路線時，同樣先顯示成功勾號 500ms，再切換到無結果狀態。
- 刷新失敗時立即移除浮層並以 Toast 提示，同時保留原結果、原更新時間與滾動位置。
- 固定浮層只用於已有結果的下拉刷新；首次點擊 `查詢` 繼續沿用現有整頁查詢狀態。
- 將路線詳情內容容器改為支援 BottomSheet 巢狀滾動的實作；內容未到頂時優先滾動內容，只有到頂後繼續下拉才允許收合或關閉彈層。
- 將新增、編輯、複製路線及臨時查詢的地點結果由獨立下拉視窗改為表單內嵌候選列表，並由共用地點輸入控制器統一管理。
- 候選列表只展示於目前聚焦的輸入框下方，高度依輸入法後方可視區動態限制為約 40%，且至少保證 3 個完整候選項目可見和可點選。
- 臨時查詢顯示候選時自動展開至接近全螢幕；一般路線表單則自動滾動，確保目前輸入框及候選列表進入可視區。
- 保留最多 100 條候選、既有搜尋 debounce、loading／無結果／失敗文案與位置、選擇校驗、交換、保存和查詢行為。
- 選擇一端地點後，如果另一端尚未選擇，保留輸入法並自動聚焦另一端；點擊空白處或第一次返回只關閉候選列表並保留輸入文字。
- 將模擬器互動與視覺驗證列為完成實作前的必要驗收項目，覆蓋長路線詳情、輸入法、候選點選及新增／編輯／複製／臨時查詢流程。

## Capabilities

### New Capabilities

無。

### Modified Capabilities

- `route-query-results-layout`: 強化路線結果下拉刷新的固定狀態反饋、成功／失敗收尾、結果替換與滾動位置行為。
- `route-detail-bottom-sheet`: 修正大量途經站展開後，詳情內容與 BottomSheet 之間的巢狀滾動及下拉收合優先級。
- `route-place-selection`: 將起終點候選改為避開輸入法的表單內嵌列表，統一聚焦、選擇、返回與外層滾動行為。

## Impact

- 主要影響 `MainActivity` 的刷新狀態管理、結果列表滾動與延遲收尾。
- 影響 `activity_main.xml` 的結果列表容器，以及刷新／成功圖示等最小必要資源。
- 影響 `RouteDetailBottomSheet` 的內容滾動容器與 BottomSheet 手勢協調。
- 影響 `PlaceInputController`、`TemporaryRouteBottomSheet`、`RouteEditActivity`、`activity_route_edit.xml` 及候選列表的最小必要 UI 資源。
- 沿用既有 `SwipeRefreshLayout`、`RecyclerView`、查詢 generation 與 `BusRouteQueryCallback.onInitialRoutes`，不改動 Citybus API、解析器、repository 查詢協議、ETA／站點預覽補全邏輯或排序規則。
- 不改動地點搜尋 API、debounce、`Place` 模型、驗證、交換、保存或查詢協議。
- 不新增第三方 UI 依賴；需補充狀態與互動測試，並在模擬器完成刷新、長內容滾動、輸入法避讓、候選點選和頁面切換驗證。
