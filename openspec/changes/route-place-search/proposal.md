## 为什么

当前路线配置只保存纯文本起点和终点，无法为后续真实巴士路线查询提供可靠经纬度参数。需要把路线管理升级为“选择地点”模式：用户输入关键词后通过 Citybus HTTP 接口搜索候选地点，从下拉列表选择后保存地点名称、纬度和经度。

## 变更内容

- 新增地点数据模型，地点由名称、纬度、经度组成。
- 新增 Citybus 地点搜索 HTTP 调用：
  - URL：`https://mobile.citybus.com.hk/nwp3/bsearch_p3.php`
  - 核心 query 参数：`q=<用户输入地点关键字>`、`timestamp=<当前毫秒级时间戳>`
  - 其他 query 参数和 headers 按用户提供的 cURL 保持，包括 Cookie。
- 解析 Citybus 返回的纯文本结果：
  - 第一行为表头，跳过。
  - `No Result` 表示无候选。
  - 正常行按 `|` 分割，第 2 列为地点名称，第 3 列为纬度，第 4 列为经度。
  - 除正常结果和无结果以外的格式视为错误。
- 路线新增/编辑页中，起点和终点不再是纯文本保存，而是可输入、可搜索、可下拉选择的地点控件。
- 用户输入 1 个字即触发自动搜索，使用 debounce 减少连续请求。
- 每次搜索最多展示接口返回的 100 条候选。
- 用户修改输入框文本后，清空已选择地点，必须重新从候选中选择。
- 保存路线时保存起点和终点的地点名称、纬度、经度；界面展示只显示地点名称。
- 起点和终点不允许完全相同。名称、纬度、经度都相同则认为是同一地点。
- 数据库升级采用删除旧路线配置并重建表的方案，避免旧纯文本数据缺少经纬度。
- 当前 Mock 巴士查询不再按起点/终点匹配，任意起点和终点都返回固定 Mock 数据。
- 后续真实巴士查询将使用起点和终点经纬度。

## 能力

### 新增能力

- `place-search-api`：通过 Citybus HTTP 接口按关键词搜索候选地点，并解析地点名称、纬度、经度。
- `route-place-selection`：在路线新增/编辑中用下拉候选选择起点和终点地点，校验必须选择候选且起终点不能完全相同。
- `route-place-storage-and-query`：将路线配置持久化为地点名称和经纬度，并让主界面和巴士查询流程使用保存的地点信息。

### 修改能力

无。

## 影响范围

- 修改路线配置数据模型和 SQLite 表结构。
- 修改数据库升级策略，升级时删除旧纯文本路线配置并重建表。
- 修改 `RouteConfigRepository` 的 insert/update/query 映射。
- 新增地点搜索模型、HTTP 仓库和纯文本解析逻辑。
- 修改路线编辑页的起点/终点输入控件和保存校验。
- 修改路线管理页和主界面的路线展示，仅展示地点名称。
- 修改 `BusRouteRepository` 接口以接收地点或经纬度参数；当前 Mock 实现固定返回测试数据。
- 需要 `INTERNET` 权限，当前 Manifest 已具备。
