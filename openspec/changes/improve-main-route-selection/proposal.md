## Why

目前主頁的查詢入口只能選擇已保存路線，當用戶偶爾需要查詢臨時起點和終點時必須先新增路線，流程過重。主頁同時也需要維持「常用路線一鍵查詢」作為最高頻操作，避免低頻臨時查詢佔用默認頁面空間。

## What Changes

- 將主頁路線入口調整為單一「常用路線」區塊，不再同時展示獨立的已選路線大卡與快捷路線小卡。
- 常用路線不足 3 條時，卡片自適應擴展寬度並佔滿可用寬度；超過 3 條時，主頁只展示 3 個快捷路線卡片與右側展開箭頭。
- 點擊常用路線卡片會選中該路線，點擊 `查詢` 使用當前選中路線發起查詢。
- 點擊右側展開箭頭打開完整常用路線列表，列表最後一項提供 `查詢臨時起點和終點` 入口。
- 臨時查詢以底部彈層展示起點、終點地點選擇與交換控件，不在主頁默認狀態常駐。
- 有效臨時查詢可保存為常用路線；保存後新路線進入常用路線列表並可被選中。
- App 在本地記錄常用路線使用次數和最近使用時間，主頁常用路線按使用頻率排序，但不在界面展示使用次數或排序規則。

## Capabilities

### New Capabilities

- `main-route-selection`: 主頁常用路線快捷卡、完整常用路線列表、臨時查詢入口和查詢來源選中行為。

### Modified Capabilities

- `route-place-selection`: 臨時查詢底部彈層也必須使用 Citybus 候選地點選擇、修改文本後重新選擇、起終點交換和有效性校驗。
- `route-place-storage-and-query`: 路線配置需要持久化使用次數和最近使用時間，並讓主頁按使用統計排序常用路線。

## Impact

- 影響主頁佈局與交互：`activity_main.xml`、`MainActivity.kt`。
- 可能新增常用路線列表/臨時查詢底部彈層佈局、適配器或對話組件。
- 需要抽取或複用現有路線編輯頁的地點搜尋、候選選中、交換和校驗邏輯，避免主頁與編輯頁重複實作。
- 影響本地資料庫結構和 `RouteConfigRepository`：新增使用次數、最近使用時間、使用記錄更新與排序讀取。
- 不改動 Citybus 路線查詢 API、ETA 查詢、結果解析、結果排序或路線詳情查詢能力。
