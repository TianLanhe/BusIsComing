## Context

路線結果卡片已從表格轉為卡片列表，並逐步加入站點預覽、下一班摘要和通知欄監控鈴鐺。近期多輪 UI 驗證顯示，卡片頂部的核心問題不在資料本身，而在左右布局邊界與垂直節奏：

- 左側站點預覽需要盡可能使用剩餘寬度，但不得侵入右側候車區。
- 右側候車區需要完整展示主候車、下一班和鈴鐺，但鈴鐺不應撐大主副標題間距。
- 卡片頂部留白看起來偏大，原因主要是頂部固定容器高度大於文字和鈴鐺實際需要，而不是外層上下 padding 不一致。
- 右側候車容器固定寬度後，容器左側空白目前也會觸發候車班次下拉，點擊熱區大於用戶視覺預期。
- 鈴鐺本身有內部 padding，若外層右 padding 仍與左側相同，視覺上會形成比左側更大的右邊距。

本設計基於已在模擬器中驗證的 XML spike。spike 僅修改 `item_bus_route.xml`，安裝到模擬器後確認 `118` 長站名卡片可以單行展示 `樂軒臺 → 海底隧道巴士轉乘站`，右側 `等候 3 分鐘`、`下一班 14 分鐘` 和鈴鐺沒有重疊。

## Goals / Non-Goals

**Goals:**

- 讓路線結果卡片頂部更緊湊，減少不必要的頂部留白。
- 讓左側路線/站點預覽和右側候車/鈴鐺成為獨立布局區域。
- 讓右側候車區採用「文字欄 + 鈴鐺欄」結構，避免鈴鐺觸控高度撐大主副標題間距。
- 讓站點預覽使用右側候車區之前的最大可用寬度，長站名單行尾部省略。
- 保留鈴鐺既有無外框樣式、圖示尺寸和 `48dp` 觸控區。
- 讓候車班次下拉只由候車文字欄觸發，右側候車容器空白不再觸發下拉。
- 讓卡片頂部右側鈴鐺/候車區的視覺邊距接近左側文字起點，同時不破壞底部分隔線和資訊區的左右邊距。

**Non-Goals:**

- 不改變 Citybus 查詢、ETA 解析、站點預覽資料來源或排序邏輯。
- 不改變通知欄監控的開啟、停止、更新或語音播報行為。
- 不重設整個結果頁視覺風格，不引入新 UI 元件庫。
- 不為本次布局調整新增運行時測量邏輯。
- 不改變路線卡片整體點擊打開詳情的行為；候車文字欄外的空白應回到卡片默認點擊語義。

## Decisions

### 右側候車區改為文字欄加鈴鐺欄

候車區保持固定寬度約束，內部從「主標題與鈴鐺同行、副標題另起一行」改為水平兩欄：左欄垂直展示 `busArrivalText` 和 `busNextArrivalText`，右欄放置 `busMonitorButton`。

理由是鈴鐺需要 `48dp` 觸控區，但主副標題本身不需要與這個高度綁定。拆成兩欄後，文字欄可以按自身內容控制行距，鈴鐺獨立垂直居中，不再讓主副標題之間出現過大的空隙。

已驗證的實作參數可作為起點：

- `busWaitArea` 寬度維持約 `160dp`。
- 鈴鐺觸控區維持 `48dp`。
- 文字欄使用剩餘寬度，主副標題右對齊。
- 主副標題間距使用約 `4dp`。

### ETA 下拉熱區綁定到候車文字欄

ETA 下拉目前由整個 `busWaitArea` 觸發，但該容器包含左側空白、候車文字和鈴鐺欄。正式實作時應讓 `busWaitArea` 回到純布局容器，新增或標記內部候車文字欄，例如 `busEtaTextColumn`，並把 `onEtaClick(route)` 綁到該文字欄。

這樣點擊「等候 N 分鐘」或「下一班 N 分鐘」附近仍可打開候車班次，但點擊候車區左側空白不再誤觸發。鈴鐺按鈕維持獨立點擊監控，卡片其他空白區域維持既有卡片點擊語義。

### 頂部區高度收斂，而非修改外層 padding

外層卡片內容上下 padding 目前同為 `12dp`，數值已對齊。視覺上的頂部空白主要來自頂部 `FrameLayout` 固定高度偏大，文字欄置中後留下內部空白。

因此本次不調整外層 `paddingVertical`，而是收斂頂部固定區高度。spike 中將頂部區和右側候車區從 `64dp` 壓到 `56dp`，鈴鐺 `48dp` 仍有上下活動空間，卡片整體更緊湊。

### 頂部右側視覺 padding 與左側文字起點對齊

卡片外層左 padding 為內容起點，但右側鈴鐺還有自身內部 padding。若外層右 padding 和左側相同，視覺上會變成「外層右 padding + 鈴鐺內部 padding」，看起來比左側更寬。

已驗證的最小策略是：讓頂部區域右側可用空間延伸到卡片內容容器右邊，依賴鈴鐺自身 padding 形成視覺右邊距；底部分隔線和底部資訊區則額外保留原本右邊距，避免底部內容貼邊。這能讓鈴鐺視覺右邊距更接近左側文字起點，並讓站點預覽獲得略長的可用寬度。

### 左側文字欄保留獨立垂直居中

左側路線和站點預覽繼續作為一個垂直文字欄，整體在頂部區中居中。這保留了用戶確認的「左右兩個區域獨立」方向，不回到兩行基線栅格。

站點預覽和路線文字都保持單行尾部省略。站點預覽可使用右側候車區左邊界之前的最大可用寬度；路線文字仍需要比站點預覽更保守，避免多段路線號靠近右側候車區。

### 驗證以真機/模擬器截圖為準

這類改動容易出現「計算看起來合理，但實際字形和系統縮放不理想」的問題。正式 apply 時需要至少做：

- `./gradlew :app:installDebug`
- 在模擬器中查詢路線結果
- 截取包含長站名、下一班和鈴鐺的卡片畫面
- 透過 UI tree 或截圖確認文本未重疊，站點預覽未侵入右側候車區

## Risks / Trade-offs

- 固定右側候車區過窄 → 兩位數候車或下一班可能省略。Mitigation: 以已驗證的約 `160dp` 作為起點，保留正常字體下完整展示兩位數候車與下一班摘要的要求。
- 頂部區壓縮過度 → 鈴鐺或文字垂直貼邊。Mitigation: 鈴鐺保持 `48dp` 觸控區，頂部區高度不得低於能容納鈴鐺和文字欄的最小高度。
- 站點預覽變長後靠近右側候車區 → 視覺上擁擠。Mitigation: 左側文字欄對右側保留固定避讓區，超出時尾部省略。
- 大字體下仍可能犧牲副標題完整性 → 這是緊湊布局的取捨。Mitigation: 優先保證不重疊、不換行、不撐高卡片；必要時讓站點預覽或下一班摘要省略。
- ETA 熱區縮小後可點範圍變小 → 用戶需要點到候車文字附近才能打開班次。Mitigation: 文字欄本身包含主副標題完整高度，保持足夠可點區域；鈴鐺和卡片詳情仍各自可點。
- 頂部右 padding 收斂後底部可能不一致 → 如果只調外層 padding，底部分隔線和資訊區會貼邊。Mitigation: 頂部區和底部區分開處理，底部分隔線與資訊區保留既有右邊距。
