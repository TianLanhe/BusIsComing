## Context

主頁路線結果卡片目前已能展示路線、候車狀態、價格、耗時、步行距離和站點預覽。`CitybusBusRouteRepository` 會先返回候選路線，再漸進式補全首程 ETA；`CitybusFirstLegEtaService` 目前負責把首程上車站 ETA 轉成候車分鐘數。

這個變更在現有查詢結果之上新增一個短時「通知欄監控」流程。用戶必須先在 App 內查詢出路線，然後手動點擊結果卡片右上角的小鈴鐺啟動監控。啟動後 App 以通知欄和語音提醒持續輔助用戶判斷何時出門。

本設計遵循 `docs/ui-style-guide.md` 的「安靜實用的現代通勤工具」風格。卡片入口使用低權重圖示按鈕，不新增一行文案；監控設定底部面板只暴露 `步行到站` 一個核心時間值，避免把提醒、停止、刷新策略變成配置頁。

## Goals / Non-Goals

**Goals:**

- 在結果卡片上提供簡潔的小鈴鐺入口，讓用戶啟動通知欄監控。
- 用底部面板確認 `步行到站` 時間，並提供互斥步速預設與可多選場景修正。
- 啟動短時前台服務，每分鐘嘗試刷新首程 ETA 並更新通知。
- 通知欄展示當前監控狀態、剩餘候車、步行到站、更新時間，以及刷新、停止、打開 App 操作。
- 根據剩餘可等待時間切換 `準備出門`、`立即出門`、`快遲到了` 三種狀態。
- 使用 Android TextToSpeech 在每次狀態切換時播報提醒文案，並允許用戶在啟動面板關閉語音。
- 監控默認允許等候 2 班車；缺少第 2 班 ETA 時使用第 1 班過站後 `+2 分鐘` 停止。

**Non-Goals:**

- 不做全天候自動監控、定時通勤計劃、桌面小組件或 App shortcut。
- 不用 WorkManager 週期任務、精準鬧鐘或後台靜默服務實現分鐘級刷新。
- 不在設定面板暴露刷新頻率、停止條件、提醒閾值或通知樣式。
- 不監控轉乘段 ETA；通知欄監控只針對結果卡片的首程上車車輛。
- 不承諾 Android 系統層面絕對每 60 秒準時刷新；產品文案使用「每分鐘嘗試更新」。

## Decisions

### 1. 使用結果卡片右上角小鈴鐺作為入口

小鈴鐺放在候車狀態右側或右上角鄰近區域，視覺圖示約 20dp 到 22dp，外層點擊區不小於 48dp。圖示採用透明背景、淡描邊或低權重 ripple，不使用實心大色塊。

理由：

- 入口與候車時間語義關聯最強。
- 不新增一行文字，不擠壓價格、耗時和步行距離。
- 比長按或滑動入口更容易發現，也更容易做無障礙描述。

替代方案：

- 長按卡片菜單：最省空間，但發現性差。
- 左滑露出監控：視覺乾淨，但 RecyclerView 滑動、誤觸和無障礙成本高。

### 2. 設定面板只暴露 `步行到站`

底部面板包含：

```text
啟動通知欄監控
82X · 上班

步行到站 [-] 4 [+] 分鐘
由接口距離、直線距離與手動調整取較長值

步行速度：慢行 3.2 / 帶小孩 3.5 / 一般 5.0 / 快走 6.0
場景修正：雨天 ×80% / 等電梯 +2 / 天橋/過馬路 +2
語音提醒：開

開始監控
取消
```

步行速度是互斥選擇；場景修正可多選。`雨天 ×80%` 只影響步行速度，不改變距離。`等電梯 +2` 和 `天橋/過馬路 +2` 是固定分鐘加成。

理由：

- 用戶真正需要確認的是「我到站要多久」。
- 步速和場景修正能快速覆蓋最常見出門變慢因素。
- 刷新頻率、停止策略、提醒閾值由 App 預設，避免面板變成偏工程化配置頁。

### 3. 步行到站取三種估算的較長值

監控 session 的 `walkingMinutes` 使用：

```text
max(接口距離估算時間, 經緯度直線距離估算時間, 用戶調整時間)
```

接口距離優先從 `getp2pstopinroute.php` 詳情中起點到首程上車站的 `步行距離(約) N米` 解析。若詳情未載入或解析失敗，該估算缺失。

經緯度直線距離使用用戶起點坐標與首程上車站坐標計算，不乘繞路係數。直線距離通常比實際步行距離短，因此只作為保底，不主動放大。

用戶調整時間是底部面板當前顯示的分鐘值，會受互斥步速預設、多選場景修正和手動 `- / +` 操作影響。

理由：

- 接口步行距離最貼近 Citybus P2P 規劃。
- 直線距離能在接口詳情不可用時提供低成本保底。
- 取較長值能避免用戶誤把步行時間調得過短而錯過車。

### 4. 前台服務負責短時監控

監控由用戶點擊 `開始監控` 後啟動前台服務。服務持有一次監控 session，包括路線、首程 ETA query、路線名稱、上車站、步行到站分鐘數、語音開關和當前狀態。

每分鐘 tick 時優先只刷新首程 ETA，不重跑完整 `ppsearch_p3.php` 點到點路線搜尋。通知 action 支持：

- `刷新`：立即嘗試刷新 ETA。
- `停止`：停止服務並移除監控通知。
- `打開 App`：返回主頁並定位當前監控路線或當前查詢結果。

理由：

- WorkManager 週期任務不適合分鐘級候車監控。
- 精準鬧鐘權限重且在省電模式下仍有調度限制。
- 前台服務有持續通知，符合用戶主動啟動、短時可見的監控語義。

### 5. 狀態判斷使用剩餘可等待時間

每次 ETA 刷新後計算：

```text
剩餘可等待 = 第 1 班 ETA 分鐘 - 步行到站分鐘
```

狀態閾值：

```text
> 2 分鐘       -> 準備出門
1 到 2 分鐘   -> 立即出門
<= 0 分鐘     -> 快遲到了
```

通知主標題展示狀態，正文展示 `82X 還有 X 分鐘 · 步行到站 Y 分鐘`。當 ETA 不可用或連續刷新失敗時，通知保留最後一次成功資料並展示資料延遲狀態；若從未成功取得 ETA，通知展示暫無候車資料並允許手動刷新或停止。

### 6. 狀態切換時使用 TextToSpeech 播報

語音提醒默認開啟。每次監控狀態切換時，服務使用 TextToSpeech 播報：

```text
準備出門：當前汽車到站剩餘 X 分鐘，請做好出門準備
立即出門：當前汽車到站剩餘 X 分鐘，請立即出門
快遲到了：當前汽車到站剩餘 X 分鐘，你要遲到了
```

為避免 ETA 抖動導致連續播報，服務應記錄上次播報狀態和時間；同一狀態不重複播報，只有狀態實際變更才播報。TextToSpeech 初始化失敗、語言不可用或播放被系統限制時，通知仍正常更新，不阻塞監控。

### 7. 停止條件內建，不暴露給用戶

停止規則：

- 用戶點擊通知 `停止` 時立即停止。
- 若有第 2 班 ETA，當第 2 班 ETA 過站後停止。
- 若沒有第 2 班 ETA，當第 1 班 ETA 過站後 `+2 分鐘` 停止。
- 若連續刷新失敗且超過保護上限，停止或提示資料延遲，避免長時間耗電。

這個變更應與 `show-multiple-citybus-eta-arrivals` 協作。若多班 ETA 尚未合併，實作時需要先完成或引入等效的第 2 班 ETA 資料模型；否則監控仍可工作，但只能使用第 1 班 `+2 分鐘` fallback 停止。

## Risks / Trade-offs

- [Risk] Android Doze、廠商省電策略或網絡不可用會讓每分鐘刷新不準時。→ Mitigation：產品文案使用「每分鐘嘗試更新」；通知展示最後更新時間和資料延遲狀態。
- [Risk] 前台服務會增加通知存在感和耗電。→ Mitigation：只允許用戶手動啟動短時 session，內建停止條件，不做全天候監控。
- [Risk] 通知權限被拒絕時無法提供核心體驗。→ Mitigation：啟動前檢查通知權限，拒絕時在 App 內提示開啟通知，且不啟動無通知的背景監控。
- [Risk] 卡片小鈴鐺擠壓候車文案或顯得突兀。→ Mitigation：限制圖示視覺尺寸，使用 48dp 隱形點擊區，窄屏和字體放大下驗證不重疊。
- [Risk] `getp2pstopinroute.php` 詳情接口較重，若每張卡都提前解析會增加延遲。→ Mitigation：只在用戶點擊鈴鐺後按需解析該路線詳情；解析失敗時降級到直線距離和用戶調整。
- [Risk] TextToSpeech 播報可能打擾用戶或初始化失敗。→ Mitigation：語音開關在面板可關閉；播報失敗不影響通知監控；釋放 TTS 資源。
- [Risk] ETA 在臨界值附近上下跳動導致狀態頻繁切換。→ Mitigation：只在狀態實際變更時播報；可在實作中加入簡單去抖，避免同一分鐘內多次播報。
- [Risk] 多班 ETA change 尚未落地會影響「等候 2 班車」停止。→ Mitigation：任務中先確認 `show-multiple-citybus-eta-arrivals` 的狀態；未可用時使用第 1 班 `+2 分鐘` fallback，並保留後續接入第 2 班 ETA 的接口。
