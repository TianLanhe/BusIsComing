## Context

主頁查詢結果目前已由表格改為 RecyclerView 卡片，並且保留顯式排序 chips、查詢狀態卡與漸入動畫。現有卡片第一行左側顯示路線、右側顯示候車，第二行展示價格、總耗時和步行距離；這和本次討論確認的方案存在三個偏差：價格仍在第一行附近形成視覺競爭，`總耗時` 文案過重，`候車暫無` 不夠貼近用戶語境。

本次設計遵循 `docs/ui-style-guide.md` 的“安靜實用的現代通勤工具”方向，保持 XML + AppCompat + Material Components 技術棧，不引入 Compose 或新依賴。

## Goals / Non-Goals

**Goals:**

- 在結果列表上方提供一次性摘要，顯示路線總數、直達數和更新時間。
- 將結果卡片收斂為可掃讀的兩層結構：第一行突出路線與等候狀態，第二行展示價格、耗時和步行距離。
- 固化已確認的文案：`耗時`、`等候 X 分鐘`、`暫無車輛`。
- 用顏色區分 ETA 狀態：可用等候使用主綠色，暫無車輛使用灰色。
- 保持排序能力不變，僅把排序控件中的 `總耗時` 改為 `耗時`。

**Non-Goals:**

- 不改動 Citybus 路線查詢 API、m1/ws 參數、HTML 解析、ETA 查詢或去重邏輯。
- 不新增路線詳情頁或 Bottom Sheet 行為；若 `add-route-detail-bottom-sheet` 後續落地，本次卡片只保留可點擊 affordance 的兼容空間。
- 不展示轉乘次數，不新增號牌樣式，不恢復表格展示。

## Decisions

### 1. 路線使用簡潔文字，不使用號牌

卡片第一行左側使用純文字展示路線序列：單程顯示 `788`，多程顯示 `789 → 15`。不使用深色號牌塊，因為號牌會佔用橫向空間，且多段路線在小屏上更容易擠壓等候狀態。

替代方案：使用號牌樣式。已放棄，因為用戶確認希望回到方案 B 的簡單文字路線。

### 2. 等候狀態固定在卡片右側

第一行右側展示 ETA 狀態：

- 可用 ETA：`等候 X 分鐘`，使用主綠色。
- 無可用 ETA：`暫無車輛`，使用灰色。
- 查詢中：保留 `候車查詢中` 或同等 loading 文案，可使用次級灰色，避免被理解為可用 ETA。

這讓用戶掃描每張卡片右側即可快速判斷是否值得等車。

### 3. 價格移入底部信息區

卡片中部增加或保留一條輕分隔線，分隔線下方按一行展示補充信息：`HK$ 8.7 · 耗時 29 分鐘 · 步行 350 米`。免費價格仍按現有格式展示為 `免費` 或既有免費展示規則，但位置同樣在底部信息區。

這樣第一行只承載路線和等候，避免價格與等候同時競爭右側核心位置。

### 4. `總耗時` 全面改為 `耗時`

卡片字段、排序 chip 和相關測試期望中，面向用戶的 `總耗時` 改為 `耗時`。底層模型字段 `durationMinutes`、排序字段 `SortField.DURATION` 和算法不需要重命名，避免無收益的代碼 churn。

### 5. 結果摘要在排序控件上方展示

查詢成功且有結果時，在結果區頂部展示：

```text
共 X 條路線，Y 條直達        更新時間：HH:MM:SS
```

`X` 取當前查詢返回的完整結果數，`Y` 取 `transferCount == 0` 的結果數。更新時間取該次查詢初始結果成功展示時的本機時間；後續 ETA 單條更新不刷新該時間，避免等候時間逐條回來時摘要跳動。

## Risks / Trade-offs

- [Risk] 多段路線文字過長時可能擠壓右側等候狀態 → 路線 TextView 使用單行省略或彈性權重，等候狀態固定在右側並保持最小可讀寬度。
- [Risk] 底部信息區三個字段在窄屏或字體放大時可能換行 → 使用可換行的水平/彈性佈局，優先保持語義完整，不允許文字重疊。
- [Risk] `暫無車輛` 灰色可能不夠醒目 → 使用與正文次級色不同但仍屬灰色系的狀態色，並通過粗體或位置維持可見性。
- [Risk] 結果摘要更新時間與 ETA 更新時間語義可能混淆 → 文案僅表示本次結果列表刷新時間，不表示每條 ETA 的官方更新時間。
