## Why

現有結果卡片右上角同時承載候車摘要和通知欄監控入口，導致右側區塊撐寬並壓縮路線與站點預覽的可讀空間。通知欄監控在 App 留在前台時可以更新，但鎖屏通知可見性、退前台或鎖屏後的定時更新，以及狀態切換語音播報都沒有達到「出門前短時盯車」的預期。

## What Changes

- 將路線結果卡片的候車摘要改為右側跨兩行資訊塊，類似合併單元格：路線號與站點預覽在左側，主候車、下一班與鈴鐺入口在右側垂直對齊。
- 保持鈴鐺現有視覺樣式、大小、透明背景和 48dp 觸控區，只調整位置，不新增外框、圓形背景或實心按鈕樣式。
- 明確鎖屏可見的監控通知要求：開始監控後通知欄和鎖屏頁面都應能看到監控通知，通知內容可展示完整路線和候車狀態。
- 強化退前台與鎖屏後的短時刷新策略：前台可使用常規 tick，退前台或鎖屏時需要使用可喚醒的系統調度機制每分鐘嘗試刷新 ETA。
- 持久化一次監控 session 的必要資料，使進程被系統回收後仍可在合理範圍內恢復監控狀態。
- 提升 TextToSpeech 可靠性：檢查語言可用性，提供繁中或中文 fallback，並在狀態切換時能在 App 退前台或鎖屏後仍嘗試播報。
- 增加語音試聽入口，用戶可在啟動監控前確認設備 TTS 是否可用。
- 保留「每分鐘嘗試更新」的產品邊界：Android 省電、Doze、網絡或用戶強制停止 App 可能導致延遲或停止，通知必須展示最後更新時間和資料延遲狀態。

## Capabilities

### New Capabilities

- `notification-monitor-reliability`: 定義通知欄監控在鎖屏可見、退前台或鎖屏後持續嘗試刷新、監控 session 恢復、語音播報可用性與系統限制降級方面的可靠性要求。

### Modified Capabilities

- `bus-route-result-table`: 調整路線結果卡片中候車摘要與通知欄監控入口的位置要求，採用右側跨兩行候車資訊塊並保持鈴鐺樣式不變。
- `route-query-results-layout`: 調整結果卡片在手機豎屏下的寬度分配、站點預覽省略和窄屏字體縮放要求，避免候車區反向撐大卡片。

## Impact

- 影響 UI：`item_bus_route.xml` 建議由線性佈局改為 `ConstraintLayout` 或等效約束佈局；`BusRouteAdapter` 需要保持候車區、鈴鐺和卡片詳情點擊語義不互相干擾。
- 影響通知：需要新增或遷移通知 channel id，設定鎖屏可見性，避免受既有 channel importance 不可變限制影響。
- 影響後台刷新：`BusMonitorService` 需要從單純 `Handler.postDelayed` 擴展為前台 tick + 系統調度喚醒的混合策略，並處理 Doze、網絡失敗和重入保護。
- 影響監控資料：需要保存和恢復 route name、首程 ETA query、步行到站時間、語音開關、上次狀態、最後成功通知文案和停止條件。
- 影響語音：需要檢查 TextToSpeech 初始化與語言設定結果，加入 fallback、試聽入口和播報失敗降級。
- 影響權限與平台能力：可能涉及 notification channel 遷移、AlarmManager、WakeLock 或相關 Android 後台限制處理；若使用精準鬧鐘能力，必須評估是否需要額外權限並避免不必要的權限擴張。
- 影響測試與驗證：需要新增卡片 layout/helper 測試、通知 channel/可見性測試、監控 session 恢復測試、調度策略單元測試、TTS fallback 測試，以及模擬器鎖屏/退前台/通知面板手動驗證。
