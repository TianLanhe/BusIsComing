## Context

主界面查询结果已经从表格迁移为路线结果卡片，用户可以看到路线、价格、总耗时、候车时间和步行距离，但无法确认具体上车站、下车站和途经站点。现有 `CitybusRouteParser` 已经从 `ppsearch_p3.php` 候选路线的 `showroutep2p(...)` 中解析首程 ETA 信息，但模型只保留首程 `FirstLegEtaQuery`，不足以支撑整条路线详情。

Citybus 网页点击路线卡片时调用 `showroutep2p(info, lid, ginfo)`，其脚本实际请求 `getp2pstopinroute.php?info=<rawInfo>&ginfo=<ginfo>&lid=<lid>&l=<lang>`。该接口返回 HTML 片段，包含每段路线的站点列表；每个站点的 `stopclick1(...)` 参数包含 `stop_id`、经纬度、站序和内部路线 variant。用户已确认使用底部弹层、途经站默认折叠、不使用 DATA.GOV.HK 详情兜底，并按 `rawInfo + lang` 缓存成功解析结果 1 天。

用户已确认路线详情采用参考图中的时间线结构，但只保留本次需要的信息：每段巴士使用不同颜色粗竖线表示，上车站和下车站作为分段端点，途经站可展开；不展示运营时间、到站时刻表、车辆到达时间、步行导航或底部操作栏。

## Goals / Non-Goals

**Goals:**

- 点击路线结果卡片后，在当前页面打开路线详情底部弹层。
- 点击后按需请求 Citybus `getp2pstopinroute.php`，不在路线列表展示时预取所有详情。
- 解析单段和多段路线的上车站、下车站、途经站、站序、站点编号和坐标。
- 站点展示名使用接口站名中逗号前的第一段。
- 路线段标题展示路线号；若接口返回方向信息，则展示 `往 XX方向`，若无可靠方向则隐藏方向文本。
- 详情 UI 使用每程分色粗竖线表达巴士分段，换乘处只用灰色虚线连接。
- 每段路线默认折叠途经站，并允许按段独立展开和收起。
- 成功解析结果按 `rawInfo + lang` 进程内缓存 1 天；失败结果不缓存。
- 详情查询失败或解析失败时展示可重试的错误状态。

**Non-Goals:**

- 不使用 DATA.GOV.HK route-stop/stop 接口重建路线详情或作为兜底。
- 不改变现有路线查询、聚合、排序和首程候车时间补全行为。
- 不在地图上绘制路线或站点。
- 不展示起点/终点步行段、换乘步行距离或换乘步行时间。
- 不展示运营时间、到站时刻表、车辆到达时间、步行导航、收藏、截图、分享、关注路线或下车提醒。
- 不新增离线持久化数据库表；缓存随 App 进程结束自然清空。

## Decisions

### 决策 1：使用 `BottomSheetDialogFragment` 展示详情

路线详情属于当前查询结果的上下文信息，不需要用户离开主查询页。使用 Material Components 的 `BottomSheetDialogFragment` 可以复用现有依赖，支持半屏到全屏滚动、关闭和加载状态，同时符合 `docs/ui-style-guide.md` 中安静实用的工具风格。

备选方案是新增详情 Activity。它导航关系清楚，但用户需要离开结果列表，返回后还要恢复列表和排序状态，交互成本更高。

### 决策 2：路线详情点击后按需请求

详情接口只在用户点击某张路线卡片后请求。这样不会拖慢路线查询首屏，也不会为用户未查看的路线消耗网络请求。弹层先展示加载状态，成功后替换为详情，失败后展示错误和重试按钮。

备选方案是在路线查询完成后预取所有详情。它能让点击后更快展示，但每次查询可能产生大量额外请求，与当前已做的路线先展示和 ETA 后台补全优化方向冲突。

### 决策 3：保存完整 P2P 详情查询元数据

新增模型用于保存 `showroutep2p(...)` 的完整参数：

```text
P2pRouteDetailQuery
├─ rawInfo
├─ generalInfo
├─ listId
└─ lang
```

同时解析 `rawInfo` 中所有 bus legs：

```text
P2pRoutePlan
└─ legs
   ├─ company
   ├─ routeVariant
   ├─ route
   ├─ boardingSeq
   ├─ alightingSeq
   ├─ bound
   └─ directionPath
```

首程 ETA 可以继续使用第一段 leg 派生出的 `FirstLegEtaQuery`，避免同一段 `showroutep2p` 字符串被两套逻辑重复、且规则不一致地解析。

### 决策 4：详情只使用 Citybus HTML 接口

详情请求 URL 使用：

```text
https://mobile.citybus.com.hk/nwp3/getp2pstopinroute.php?info=<rawInfo>&ginfo=<generalInfo>&lid=<listId>&l=<lang>
```

`ssid` 和 `sysid` 不作为必需参数；如果真实设备验证发现接口在某些环境要求会话参数，再复用现有 Citybus 首页初始化方式补齐。详情失败时不调用 DATA.GOV.HK 作为兜底，直接展示错误状态。

备选方案是使用 DATA.GOV.HK 的 `route-stop` 和 `stop` JSON API 重建详情。该方案结构化更稳定，但需要更多请求，并且与 Citybus 点到点页面展示的内部 route variant、换乘提示不完全等价。

### 决策 5：按 `rawInfo + lang` 做 1 天进程内缓存

缓存 key 使用完整 `rawInfo` 和 `lang`，因为同一条 P2P 方案在不同语言下站名不同。缓存值为结构化后的 `RouteDetail`，TTL 为 1 天。只缓存成功解析结果，失败、空响应或解析异常不缓存，避免短暂网络或接口异常污染后续点击。

### 决策 6：用 `rawInfo` 的 seq 判定站点角色

详情 HTML 中存在 `list_board.png`、`list_alive.png` 等视觉标记，但解析时不应只靠图片名判断上下车。系统应以 `rawInfo` 中每段 leg 的 `boardingSeq` 和 `alightingSeq` 作为角色判定依据：等于上车站序为上车站，等于下车站序为下车站，位于中间的为途经站。

详情 HTML 负责提供站名、站点编号和坐标；`rawInfo` 负责提供路线段边界和角色。

### 决策 7：路线详情使用轻量时间线 UI

底部弹层成功态顶部展示路线摘要，正文使用纵向时间线展示每段巴士。每段包含路线号、可选方向、上车站、下车站和途经站折叠控件；每段左侧使用不同颜色的粗竖线，颜色只用于区分路线段，不表达实时状态。

站点名称展示时取接口站名中第一个逗号前的文本，例如 `樂軒臺, 柴灣道` 展示为 `樂軒臺`。这可以减少站点后缀、道路名或补充描述造成的换行和视觉噪音；完整原始站名仍可保留在结构化模型中，便于后续调试或扩展。

路线段方向只在接口返回可靠方向时展示，格式为 `往 XX方向`；如果方向缺失或无法确认，隐藏方向文案，不根据站名自行推断。

换乘位置只用灰色虚线或留白连接两段巴士，不展示换乘步行距离或时间。当前详情接口目标是确认上下车和途经站，步行距离/时间来自点到点搜索摘要而非稳定分段详情，放入弹层会制造不可靠信息。

弹层不展示参考图中未纳入本次需求的运营时间、到站时刻表、车辆到达时间、步行导航、收藏、截图、分享、关注路线和下车提醒。

## Risks / Trade-offs

- [Risk] Citybus `getp2pstopinroute.php` 返回的是 HTML，结构可能变化。→ Mitigation：解析器集中在单独组件，使用真实 HTML fixture 覆盖单段、多段和用户提供的 `118` 样例；解析失败只影响详情弹层，不影响路线列表。
- [Risk] 部分 `ppsearch_p3.php` 候选路线缺少 `showroutep2p(...)` 元数据。→ Mitigation：路线结果仍展示，点击详情时显示“路线详情暂不可用”。
- [Risk] 用户频繁点击多条路线可能发起多个详情请求。→ Mitigation：只允许当前弹层请求更新当前内容；关闭弹层或点击其他路线后忽略旧请求结果；成功结果缓存 1 天。
- [Risk] 进程内缓存不能跨 App 重启。→ Mitigation：本次以降低单次使用过程中的重复点击请求为目标，不引入持久化存储复杂度。

## Open Questions

无。用户已确认：使用底部弹层；默认折叠并按段独立展开；站名只展示逗号前第一段；每段巴士使用分色粗竖线；接口返回方向时展示 `往 XX方向`，否则隐藏方向；换乘只用灰色虚线连接；不使用 DATA.GOV.HK 兜底；按 `rawInfo + lang` 缓存 1 天；弹层先加载、失败可重试、失败不缓存。
