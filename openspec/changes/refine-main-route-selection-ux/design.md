## Context

現行 `main-route-selection` 已完成主頁常用路線快捷卡、完整列表、臨時查詢底部彈層和使用次數排序。主頁目前以 `routeConfigs` 的排序結果直接取前三條作為快捷卡，並用右側獨立 `>` 按鈕打開完整列表。

這個基礎流程可以保留，但需要補齊兩類查詢上下文：
- 常用路線：用戶選中一條不在原始 Top 3 的路線後，主頁仍要清楚展示「當前選中的是哪條」。
- 臨時查詢：用戶完成臨時起終點查詢後，結果區要用極小空間說明結果基於哪組起終點。

本變更只調整展示與交互，不改變路線配置資料模型、使用次數排序規則、Citybus 查詢或結果解析。

## Goals / Non-Goals

**Goals:**
- 讓當前選中的常用路線在主頁快捷卡中始終可見且明確。
- 將選中態控制在明顯邊框與淺色背景，不加入勾選圖示或 `目前` 標籤。
- 將右側獨立 `>` 展開按鈕替換為標題行右側 `全部` 文字入口。
- 為臨時查詢結果提供單行、低高度、可保存的上下文條。
- 保持常用路線查詢結果區乾淨，不額外增加 `目前查詢` 摘要。
- 移除排序 chips 上方獨立的 `排序` 標題文字，讓結果區更緊湊。

**Non-Goals:**
- 不新增手動置頂、拖拽排序、分組或收藏標籤。
- 不改變 `usage_count`、`last_used_at` 的持久化與排序規則。
- 不將臨時查詢輸入框常駐主頁。
- 不改動 Citybus HTTP 請求、ETA 更新、結果排序或結果詳情。

## Decisions

### Decision: 快捷卡可見列表由「原始 Top 3」與「當前選中」共同決定

主頁渲染快捷卡時先保留 `routeConfigs` 的真實排序作為 `sortedRoutes`，再計算展示列表：

```text
baseTop3 = sortedRoutes.take(3)

if saved route count <= 2:
    visibleRoutes = sortedRoutes
else if selectedRoute is null or selectedRoute in baseTop3:
    visibleRoutes = baseTop3
else:
    visibleRoutes = [selectedRoute] + baseTop3 excluding selectedRoute, limited to 2 fillers
```

這代表「臨時提升」只存在於 UI 展示層，不寫回資料庫，不修改 `routeConfigs` 順序，也不影響完整列表排序。當用戶再選中另一條非原始 Top 3 路線時，新的 `selectedRoute` 重新參與上述計算，上一條自然回到原本排序位置。

替代方案：
- 將選中路線放到第三位。這保留前兩個最高頻入口，但第三位視覺權重較弱，且靠近更多入口時容易被理解為普通候選，不利於建立「當前選中」信心。
- 不臨時提升，只靠完整列表選中態。這會讓用戶回到主頁後看不到剛選中的路線，無法解決核心問題。

### Decision: 選中態只使用邊框與淺色背景

選中快捷卡使用較明顯的品牌色邊框和淡品牌色背景；未選中卡片保持普通背景和弱邊框。卡片內仍只展示路線名稱與 `起點 -> 終點`，不加勾選圖示、`目前` 標籤、排名、使用次數或排序說明。

原因：
- 選中態需要足夠可見，但不應讓卡片變成狀態說明卡。
- 邊框加背景能兼顧觸達性與頁面克制感。

### Decision: 使用標題行 `全部` 取代右側獨立箭頭

`常用路線` 標題改為左右佈局：左側標題，右側小文字入口 `全部`。點擊 `全部` 打開完整常用路線列表，列表底部保留 `查詢臨時起點和終點`。

`全部` 入口建議在存在保存路線時可見，即使只有 1-2 條路線也可打開完整列表。這樣臨時查詢入口仍可從常用路線上下文中到達，不需要在首頁額外常駐另一個文字入口。

替代方案：
- 只靠橫向滑動展示更多路線。視覺最輕，但發現性不足，不能作為唯一入口。
- 使用末尾 `更多` 小卡。語義清晰，但會佔用一個快捷路線卡位，與「展示 3 條快捷路線」方向衝突。

### Decision: 臨時查詢上下文屬於結果區，不屬於常用路線區

臨時查詢發起後，結果區上方顯示一條單行上下文條：

```text
[臨時] 興華邨2豐興樓 -> 北角衛理堂                    保存
```

上下文條高度應接近 chip 或輔助行，起終點文字單行省略，不換行。它在臨時查詢的 loading、成功、失敗和無結果狀態中保持可見，直到用戶切換常用路線、清空結果或發起常用路線查詢。

`保存` 操作使用與臨時查詢底部彈層相同的保存語義：要求路線名稱、做重複校驗，保存成功後刷新常用路線列表並選中新路線。

原因：
- 臨時查詢結果必須說明基於哪組起終點，否則用戶容易忘記剛查的是什麼。
- 上下文只在臨時查詢後出現，不增加常用查詢默認負擔。
- 放在結果區而不是常用路線區，可以避免把臨時路線誤認為已保存常用路線。

### Decision: 查詢結果綁定發起時的查詢上下文快照

主頁應明確記錄當前結果對應的查詢來源，例如：

```kotlin
sealed class QueryContext {
    data class Saved(val routeId: Long) : QueryContext()
    data class Temporary(val originName: String, val destinationName: String, val origin: Place, val destination: Place) : QueryContext()
}
```

發起查詢時建立上下文快照，progressive callback 仍通過現有 `querySequence` 忽略舊回調。切換常用路線、清空結果或開始常用路線查詢時隱藏臨時上下文條。

### Decision: 排序控件不再使用獨立標題

結果區排序控件直接以 chips 展示可選排序項，例如 `路線`、`價格`、`耗時`、`候車`、`步行`。不在 chips 上方額外展示 `排序` 文字。

原因：
- 這組控件本身的文案已經足夠表達排序語義，獨立標題信息量低。
- 移除標題可以節省結果區垂直空間，抵消臨時查詢上下文條帶來的少量高度成本。
- 視覺上讓結果區從「標題 + 控件 + 摘要 + 列表」變為更直接的「控件 + 摘要 + 列表」。

## Risks / Trade-offs

- [Risk] `全部` 在只有 1-2 條常用路線時看起來像冗餘入口。→ Mitigation: 使用輕量文字按鈕，並讓完整列表同時承載臨時查詢入口，使其有實際用途。
- [Risk] 選中路線臨時提升到第一張後，前三張快捷卡不再完全等同使用排序 Top 3。→ Mitigation: 僅在選中路線不在原始 Top 3 時觸發，完整列表仍保持真實排序，UI 不外露排序規則。
- [Risk] 臨時上下文條增加結果區一行高度。→ Mitigation: 僅臨時查詢後顯示，單行省略，高度控制在 chip/輔助行級別。
- [Risk] 移除 `排序` 標題後，首次用戶可能需要從 chips 文案理解用途。→ Mitigation: 保留清晰的排序項文字和當前排序方向，並保持 chips 位於結果列表上方。
- [Risk] 從結果上下文條保存臨時路線會與底部彈層保存流程重複。→ Mitigation: 抽取或復用保存臨時路線的命名、校驗和 repository 寫入邏輯，避免兩套行為分叉。
