## Context

BusIsComing 的主頁目前以快速查詢為核心，`activity_main.xml` 在普通狀態頂部展示 `乘車碼` 與 `管理路線` 兩個文字按鈕；首次引導狀態僅展示低權重 `乘車碼`。`常用路線` 標題行右側已有 `全部` 文字入口，點擊後打開完整常用路線列表。臨時查詢結果上下文條目前展示起終點路徑和 `保存` 入口，但沒有就地編輯目前臨時查詢起終點的入口。這套結構在只有路線管理和查詢能力時足夠，但新增關於我們、隱私政策、分享、反饋、語言、評分和更新後，若繼續把入口平鋪到首頁，會破壞 `docs/ui-style-guide.md` 定義的「安靜實用的現代通勤工具」定位。

本 change 將 App 級低頻入口收斂到獨立 `設定` 頁，同時把 `管理路線` 保留在常用路線工作流內，並補齊臨時查詢結果後的編輯再查路徑。實作應沿用 XML + AppCompat + Material Components，不引入 Compose、Play Core 或新的後端服務。

## Goals / Non-Goals

**Goals:**

- 在普通主頁與首次引導頁提供清楚但克制的右上角 `設定` 圖示入口。
- 新增獨立 `設定` 頁，按 `偏好`、`支援`、`關於` 分組展示已確認入口。
- 新增設定內 `關於我們` 二級頁，展示 App 名稱、版本、簡介和官網入口。
- 讓 `隱私政策`、官網、分享應用和問題反饋使用系統外部能力，並在無可處理目標時以 Toast 降級。
- 保留 `全部` 作為完整常用路線入口，新增 `管理` 文字入口，兩者同在 `常用路線` 標題行且不互相貼近。
- 在臨時查詢結果上下文條中新增 `編輯` 入口，讓用戶帶入目前臨時查詢起終點，修改後繼續臨時查詢。
- 除新增臨時查詢編輯入口外，保持既有路線查詢、常用路線選擇、首次引導、排序、刷新與通知監控流程不變。

**Non-Goals:**

- 不實作真正多語言切換，不新增 AppCompat locale 或資源多語言切換。
- 不實作應用評分、Google Play In-App Review、檢查更新、Play Core In-App Updates 或自建更新服務。
- 不修改 Citybus、DATA.GOV.HK、Google reverse geocoding 或本機資料 schema。
- 不把 `管理路線` 放入 `設定` 頁，不新增側邊抽屜、底部導航或底部「更多」面板。
- 不重新設計路線管理頁、路線編輯頁或結果卡片。
- 不把臨時查詢編輯做成保存常用路線流程；不新增臨時路線名稱、不自動保存、不更新常用路線使用統計。
- 不修改 Citybus 查詢參數、地點搜尋 API、Google reverse geocoding 或 ETA 補齊流程。

## Decisions

### Decision 1: 使用獨立設定頁，而不是抽屜、底部導航或底部彈層

做法：主頁右上角新增白底描邊圓形 `設定` 圖示，點擊後啟動獨立設定 Activity。設定頁使用與現有頁面一致的淡綠到近白根背景、白色實體卡片、克制分組標題和左上返回入口。系統返回鍵與左上返回都返回主頁。

原因：這些入口大多是 App 級低頻支援或資訊披露，不是和首頁查詢同級的目的地。獨立頁能容納隱私政策、版本、暫不支援狀態與後續多語言 URL 擴展，也不佔用結果列表或首頁第一屏空間。

替代方案：

- 左側導航抽屜：擴展性強，但會把 App 表達成多主目的地結構，對目前單主任務通勤工具偏重。
- 底部導航 `首頁 / 設定`：設定不是和首頁同等重要的高頻目的地，且會長期佔用底部空間。
- 底部更多面板：適合短操作清單，但本功能包含二級關於頁、外部連結與未來語言狀態，長期會變擁擠。

### Decision 2: 設定頁使用固定分組與短文案

做法：設定頁按以下順序展示：

```text
設定

BusIsComing
版本 x.x.x

偏好
- 語言

支援
- 分享應用
- 問題反饋
- 應用評分
- 檢查更新

關於
- 關於我們
- 隱私政策
```

原因：`語言` 是偏好；分享、反饋、評分、更新是支援／營運類動作；關於與隱私是資訊披露。短文案可降低窄屏和字體放大下的換行風險。

替代方案：把所有入口放成單一列表。否決原因是語義混雜，後續增加入口時更像雜項頁。

### Decision 3: 本期保留部分入口為可點擊占位

做法：`語言`、`應用評分`、`檢查更新` 保持正常可點擊樣式，但點擊後 Toast 提示：

- `暫不支援語言切換`
- `暫不支援應用評分`
- `暫不支援檢查更新`

原因：用戶已確認這些功能後續再細化，但希望入口先存在。保持正常樣式可避免之後接入真實能力時出現大幅視覺變更。

替代方案：置灰禁用。否決原因是不可點擊狀態難以說明後續規劃，也和用戶確認的「点击弹 Toast」不一致。

### Decision 4: 外部動作走系統 Intent 並以短 Toast 降級

做法：

- `分享應用` 使用系統分享 Intent，內容為：
  `搭巴士前想快一點知道哪條路線合適？BusIsComing 可以保存常用路線，快速比較候車時間、票價和耗時。https://www.busiscoming.com`
- `問題反饋` 使用郵件 Intent，收件人 `hezhenyu966@gmail.com`，主題 `BusIsComing 問題反饋`，正文預填 App 版本、Android 版本、設備型號和問題描述區。
- `隱私政策` 打開 `https://www.busiscoming.com/zh-hant/privacy/`。
- `關於我們` 頁的官網入口打開 `https://www.busiscoming.com`。
- 打開失敗時分別 Toast：`暫時無法分享應用`、`暫時無法開啟問題反饋`、`暫時無法開啟隱私政策`、`暫時無法開啟網站`。

原因：分享、郵件和外部網頁都已有成熟系統能力，不需要新增後端或自定義 WebView。隱私政策使用線上 URL 也能和正式分發所需的 privacy policy URL 保持同一來源。

替代方案：內建 WebView 或本地隱私文本。否決原因是增加維護面，且本地內容容易和線上政策不同步。

### Decision 5: 關於我們使用設定內二級頁

做法：`關於我們` 打開設定內二級頁，左上返回到 `設定`。頁面展示 App 名稱、版本、簡介 `保存常用路線，出門前快速比較巴士方案。`、官網 URL 入口。

原因：關於頁雖然短，但包含版本與可點擊官網，後續可能增加資料來源或版權資訊。普通二級頁比彈層更一致，也便於處理狀態欄、字體縮放和返回。

替代方案：使用 AlertDialog 或 BottomSheet。否決原因是可擴展性弱，且會和設定頁其他列表入口的頁面語義不一致。

### Decision 6: 管理路線留在常用路線區，使用兩個低權重文字入口

做法：主頁普通狀態頂部保留 `乘車碼` 和右上角 `設定`；不再放頂部 `管理路線` 大按鈕。`常用路線` 標題行右側保留 `全部` 文字入口，旁邊新增獨立 `管理` 文字入口。`全部` 和 `管理` 之間保留清楚間距，各自具備至少接近 40dp 到 48dp 的觸控區，`管理` 入口提供無障礙描述 `管理路線`。

原因：`管理路線` 是路線查詢工作流的一部分，不是 App 設定；但它也不應繼續和 `乘車碼` 並列為頂部大按鈕。`全部` 和 `管理` 都使用低權重文字入口，能節省高度，維持現有 `常用路線 ... 全部` 的緊湊節奏，同時讓兩個入口都是一步操作且可辨識。

替代方案：

- `全部` 和 `管理` 兩個文字按鈕同列：窄屏下容易顯得貼近，誤觸風險更高。
- 標題下方雙操作列：觸控清楚但多佔一行，和用戶對首頁空間的要求衝突。
- 把 `管理路線` 放入設定頁：概念錯位，使用者想整理常用路線時不一定會進設定。

### Decision 7: 臨時查詢上下文條提供保存與編輯兩個低權重入口

做法：臨時查詢結果上下文條保留 `臨時` 標籤和起終點路徑，右側展示 `編輯` 與 `保存` 兩個文字入口或等效低權重入口。`編輯` 點擊後打開既有 `TemporaryRouteBottomSheet`，並用目前 `QueryContext.Temporary` 的 origin／destination 預填起點和終點輸入框及 selected place 狀態。用戶修改後點 `使用此路線查詢` 才發起新臨時查詢；關閉彈層或返回不覆蓋現有結果、排序、更新時間或臨時上下文。直接從首次引導頁或其他入口打開臨時查詢時，仍沿用既有空表單／目前位置自動填入行為；預填編輯模式 SHALL NOT 被自動目前位置流程覆蓋。

原因：`保存` 與 `編輯` 都是臨時查詢結果後的自然後續動作，放在上下文條內比回到首頁入口更直接。復用既有底部彈層和地點候選控制器可保留 Citybus 地點搜尋、校驗、交換、定位與保存邏輯，避免新增一套表單。

替代方案：

- 在結果上下文條只保留 `保存`，用戶重新點 `直接查詢一次`。否決原因是入口距離當前結果太遠，也丟失了目前起終點作為編輯起點。
- 新增專用全屏臨時查詢編輯頁。否決原因是臨時查詢本身已由底部彈層承載，新增頁面會擴大範圍並造成兩套相似表單。
- 點擊路徑文字直接進入編輯。否決原因是可發現性和無障礙語義弱，且容易和查看結果或路線詳情的意圖混淆。

## Risks / Trade-offs

- [Risk] 新增設定入口與管理入口可能在窄屏或字體放大時擠壓 `常用路線` 標題。→ Mitigation：使用短文案、固定最小觸控區、標題單行省略或必要時保持足夠間距；在窄屏和字體縮放場景驗證不重疊。
- [Risk] `全部` 與 `管理` 同為文字入口後可能視覺貼近。→ Mitigation：兩者保持明確 margin，並在 UI 測試或人工檢查中確認觸控和語義。
- [Risk] 外部瀏覽器、分享或郵件 App 不可用。→ Mitigation：所有外部 Intent 都捕捉失敗並顯示對應短 Toast，不阻塞當前頁。
- [Risk] 隱私政策 URL 後續因多語言而需要變更。→ Mitigation：將 URL 作為資源或集中常量，不散落在點擊邏輯；本期固定繁體 URL，後續多語言 change 可接入 locale-based URL provider。
- [Risk] 設定頁新增後與既有頁面風格不一致。→ Mitigation：沿用 `docs/ui-style-guide.md`、現有顏色、8dp 左右圓角、白色實體卡片和淡綠根背景；不引入抽屜、底部導航或大面積新視覺語言。
- [Risk] 暫不支援入口讓用戶期待已完成能力。→ Mitigation：點擊立即 Toast 提示暫不支援，規格明確本期不接真實評分、更新或語言切換。
- [Risk] 臨時上下文條同時放 `編輯` 與 `保存` 後在窄屏擠壓起終點路徑。→ Mitigation：兩個入口使用短文字低權重按鈕、固定最小觸控區，路徑單行省略；在窄屏和字體放大下檢查不重疊。
- [Risk] 預填編輯模式可能被目前位置自動填入覆蓋。→ Mitigation：`TemporaryRouteBottomSheet` 接受可選初始起終點；有初始起點時不啟動自動目前位置填入，只保留手動定位按鈕。
- [Risk] 關閉編輯彈層可能誤清空當前臨時結果。→ Mitigation：只有點擊 `使用此路線查詢` 且通過校驗後才更新 `QueryContext.Temporary` 和結果；dismiss/back 不改變現有上下文。

## Migration Plan

1. 在主頁新增設定入口並調整 `管理路線` 入口位置，保留既有路線選擇和管理頁目的地。
2. 新增設定頁和關於頁，註冊 Activity，集中定義 URL、分享文案、反饋郵箱和 Toast 文案。
3. 接入分享、反饋、外部 URL 打開和占位 Toast 行為。
4. 在臨時查詢上下文條新增 `編輯` 入口，並讓 `TemporaryRouteBottomSheet` 支援可選起終點預填。
5. 補充 contract／UI 測試與人工截圖驗證。
6. 若發現新入口影響首頁空間，可回退為只恢復舊 `管理路線` 頂部按鈕、移除設定 Activity，並移除臨時上下文條 `編輯` 入口；不涉及資料遷移。
