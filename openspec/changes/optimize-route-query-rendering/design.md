## Context

当前主界面点击查询后，`MainActivity` 在单线程 `queryExecutor` 中调用 `BusRouteRepository.searchRoutesProgressively(...)`。`CitybusBusRouteRepository` 会先并发请求 Citybus `m1=T/F/W` 并聚合路线，再优先等待前 5 条路线 ETA 完成或达到 5 秒超时，之后才通过 `onInitialRoutes` 让界面展示列表。

实测显示，路线查询接口和 ETA 相关接口都存在明显慢尾。首屏结果被 ETA 阶段阻塞时，用户会长时间停留在“正在查询路线”状态，即使路线列表本身已经可以展示。另一个风险是：首屏展示后，旧查询的剩余 ETA 补全仍在同一次查询任务中继续执行；用户再次点击查询时，新查询可能排队等待旧 ETA 收尾。

## Goals / Non-Goals

**Goals:**

- 路线聚合完成后立即展示列表，候车时间以后台渐进方式补全。
- 新查询、切换路线或离开页面时，旧 ETA 任务不再污染界面，也不应阻塞新查询开始。
- 保持既有 Citybus 查询参数、三种 `m1` 聚合、去重、价格、耗时、步行距离和排序能力不变。
- 保持 UI 现有查询中、无结果、失败和结果卡片体验，只调整加载结束时机和 ETA 更新路径。

**Non-Goals:**

- 不做 `m1=T/F/W` 渐进合并；本次仍等待三种 `m1` 路线结果聚合完成后展示。
- 不引入新的 HTTP 客户端或协程依赖。
- 不实现 route-stop 持久化缓存；现有进程内缓存策略保持不变。
- 不改造结果卡片视觉样式。

## Decisions

### Decision 1: 路线列表先渲染，ETA 全后台补全

路线查询成功、聚合和默认排序完成后，Repository 立即回调 `onInitialRoutes`。初始列表中，有可解析首程 ETA 信息的路线显示为查询中；无法解析首程 ETA 的路线显示为不可用。随后后台 ETA 任务逐条完成并通过 `onRouteWaitTimeUpdated` 更新对应路线。

这样可以把首次可见结果的等待时间控制在 Citybus 路线查询和本地解析聚合之后，不再叠加 ETA 的 0 到 5 秒等待。

备选方案是把首屏 ETA 等待时间从 5 秒缩短到 500 到 1000 毫秒，但这仍会让列表渲染依赖 ETA，且网络慢尾下收益不稳定。

### Decision 2: 路线查询任务和 ETA 补全任务分离

`searchRoutesProgressively` 不应在发出初始路线回调后继续同步等待所有 ETA 完成。路线查询任务负责完成 ppsearch、解析、聚合和初始回调；ETA 补全任务使用独立、有上限的后台执行器或任务集合继续运行。

这样用户再次点击查询时，新路线查询不会因为上一轮剩余 ETA 的 future 仍在等待而排队。

备选方案是在原单线程查询任务中继续跑 ETA，但通过 `querySequence` 忽略旧更新。这个方案只能避免旧结果污染 UI，不能解决新查询被旧 ETA 阻塞的问题。

### Decision 3: 新查询触发时取消旧 ETA 任务并保留序列校验

主界面发起新查询、切换已保存路线或销毁页面时，应取消旧查询关联的 ETA 后台任务。即使底层 HTTP 请求无法立刻中断，完成后的回调仍需通过查询序列或 token 校验，确保旧 ETA 更新不会应用到当前结果。

备选方案是只保留序列校验不做取消。该方案实现简单，但会继续占用线程和网络请求，用户连续查询时仍会浪费资源。

### Decision 4: ETA 更新按完成顺序发出

后台 ETA 结果应在单个请求完成后尽快回调，而不是按提交顺序逐个 `get()`。否则某个较慢 ETA 会挡住后面已经完成的 ETA 更新。

可以通过 completion queue、按任务完成回调或等价机制实现；具体代码形态以当前 Java/Kotlin executor 风格为准。

## Risks / Trade-offs

- 首屏列表展示时，候车时间更可能短暂显示为“查询中” -> 通过明确的查询中状态和单条渐进更新降低困惑。
- ETA 任务取消依赖 `HttpURLConnection` 的中断表现，底层请求可能不会瞬间停止 -> 保留序列校验作为最终防线，并在任务完成后丢弃旧结果。
- 路线查询任务和 ETA 任务分离会增加生命周期管理复杂度 -> 将取消入口集中在新查询、清空结果和页面销毁路径，避免散落状态。
- 如果用户当前按候车时间排序，ETA 后台更新会触发列表重排 -> 保持既有排序规则，只在当前排序字段为候车时按最新状态刷新。

## Migration Plan

1. 调整 Repository 渐进查询流程：路线聚合后立即回调初始结果。
2. 将 ETA 补全从同步等待改为独立后台任务，并支持取消旧任务。
3. 保留 `querySequence` 或等价 token 校验，确保旧 ETA 更新不会写入当前 UI。
4. 更新现有渐进 ETA 单元测试，增加首屏不等待 ETA、新查询不被旧 ETA 阻塞、旧 ETA 取消或忽略的覆盖。
5. 运行 `./gradlew build` 验证。
