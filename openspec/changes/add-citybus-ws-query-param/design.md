## Context

当前 App 已经实现 Citybus `ppsearch_p3.php` 路线查询，并在仓库层并发请求 `m1=T`、`m1=F`、`m1=W` 三种模式，再按路线号序列、总价格、总耗时和步行距离去重。上一轮真实验证发现，不带 `ws` 参数时三种 `m1` 可能都只返回 `788`、`780` 两条直达路线。

用户进一步确认，路线数量差异只和 `ws` 参数有关。根据用户提供的样例，增加 `ws=1.3` 后：

- `m1=T` 返回最快路线集合，包含 `789 → 619`、`789 → 15`、`788`、`780`。
- `m1=F` 返回最便宜路线集合，包含 `8X → 10`、`8X → 1`、`780`、`788`，其中 `8X → 10` 和 `8X → 1` 的第二段为 `免費 *`。
- `m1=W` 返回步行距离较短路线集合，包含 12 条候选，如 `8 → 90`、`8 → E11A`、`8 → 10`、`788`、`780`。

本次变更仍使用 Kotlin + XML、`HttpURLConnection`、jsoup 和仓库层聚合，不引入新的网络框架、缓存策略或 UI 控件。

## Goals / Non-Goals

**Goals:**

- 在 `CitybusBusRouteRepository.buildRouteUrl()` 构造的每个路线查询 URL 中固定加入 `ws=1.3`。
- 三种 `m1` 请求都携带同一个 `ws=1.3`，并继续共用同一个查询时间 `t`。
- 继续不新增 `loc`、`ssid`、`sysid`。
- 保持三模式并发、部分成功、聚合去重、默认总耗时升序和五列表格展示行为。
- 用用户给出的 T/F/W HTML 样例补足单元测试，确保加 `ws=1.3` 后的多程、免费价格、步行距离和去重都被覆盖。
- 重新进行真实接口验证，对比“不带 `ws`”和“带 `ws=1.3`”在同一查询条件下的返回路线数量和代表路线。
- 记录 `ws=1.3` 下三种 `m1` 的返回差异和合并后的展示结果。
- 制定覆盖 URL、解析、聚合、异常、UI 和模拟器端到端的测试计划。

**Non-Goals:**

- 不新增 `loc`、`ssid`、`sysid` 或 Citybus 网页会话初始化流程。
- 不改变 `m1=T/F/W` 的枚举集合和并发策略。
- 不改变去重键。
- 不改变预计汽车到站时间的阶段性规则，仍暂时使用总耗时。
- 不新增用户可配置 `ws` 的入口。

## Decisions

### 决策 1：把 `ws=1.3` 作为固定查询参数写入 URL 构造层

`ws=1.3` 是 Citybus 点到点路线搜索接口的固定参数，应由 `CitybusBusRouteRepository.buildRouteUrl()` 统一添加。这样每个 `m1` 请求自然携带同一参数，UI 和调用方不需要了解该网页接口细节。

备选方案：

- 在 `searchRoutes()` 循环中手动拼接 `ws`：会让 URL 构造逻辑分散，测试更难覆盖。
- 把 `ws` 作为可配置字段传入仓库：目前没有用户配置场景，会增加不必要的状态面。

### 决策 2：只新增 `ws=1.3`，继续排除 `loc`、`ssid`、`sysid`

用户已确认本轮只需要新增 `ws=1.3`。虽然网页完整请求中还会出现 `loc`、`ssid`、`sysid`，但这些参数会带来会话依赖和稳定性风险，不纳入本轮实现。

测试应明确验证：

- URL 包含 `ws=1.3`。
- URL 不包含 `loc=`、`ssid=`、`sysid=`。
- 三种 `m1` 请求的 `ws` 值一致。

### 决策 3：用样例 HTML 测试真实差异，不依赖实时接口返回稳定性

Citybus 实时返回会受时间、班次和服务状态影响，因此自动化测试不应只依赖实时接口。用户提供的 T/F/W 样例更适合作为 deterministic fixture：

- W fixture：断言解析出 12 条路线，并覆盖最短步行距离 `95`。
- F fixture：断言解析出免费转乘路线，`8X → 10` 总价为 `8.1`、步行距离 `355`。
- T fixture：断言解析出最快多程路线，`789 → 619` 总价为 `16.4`、总耗时 `28`、步行距离 `363`。
- 聚合 fixture：T/F/W 三组样例合并后按当前去重键应得到 16 条唯一路线。

### 决策 4：保持当前默认排序和 UI 展示不变

本次变更只改变查询参数和验证覆盖，不改变表格行为。聚合结果仍按总耗时升序作为默认展示，表头仍显示总耗时升序状态。

### 决策 5：真实验证先做带/不带 ws 的 cURL 对照，再做模拟器端到端

真实验证应拆成两层：

- cURL 对照：用同一组坐标、同一个查询时间和同一个 `m1`，分别请求不带 `ws` 和带 `ws=1.3` 两个 URL；记录 HTTP 状态码、响应大小、候选路线数量和代表路线，证明 `ws=1.3` 返回更完整。
- 三模式 cURL：在带 `ws=1.3` 的请求中分别请求 `m1=T/F/W`，记录三种模式返回的候选路线数量和代表路线，确认三者存在差异。
- 模拟器端到端：在 Pixel 8 中准备 `漁灣邨漁進樓 -> 港鐵中環站`，点击查询后验证 UI 展示合并结果、多程路线、免费累计价格、步行距离列、默认排序和横向滚动。

## Risks / Trade-offs

- [Risk] Citybus 后续改变 `ws` 含义或版本值。→ Mitigation：把 `ws=1.3` 作为常量集中定义，并在测试中明确覆盖 URL 参数；后续如需改值只改一处。
- [Risk] 实时接口返回路线数量随时间变化。→ Mitigation：自动化测试使用用户样例 fixture；真实验证记录时间、请求 URL 参数和代表路线，不把实时数量作为唯一可靠证据。
- [Risk] 新增 `ws` 后返回更多候选，UI 列表更长。→ Mitigation：现有 RecyclerView 和横向滚动表格可承载更多行；模拟器验证列表滚动和排序。
- [Risk] 三种模式返回同一路线但不同耗时/步行距离。→ Mitigation：继续使用用户确认的四字段去重键，避免误合并。
- [Risk] 免费价格和普通价格混合样例更多。→ Mitigation：用 F fixture 覆盖 `免費 *`，用 W/T fixture 覆盖普通多段价格累计。

- 用 cURL 对比不带 `ws` 和带 `ws=1.3` 的真实返回差异。
- 用 cURL 重新验证 `ws=1.3` 下三种 `m1` 返回差异。
- 在 Pixel 8 模拟器中使用指定路线验证合并后 UI 结果。
- 更新本 change 的 `tasks.md` 验证记录。
