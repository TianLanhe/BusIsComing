## Context

當前路線卡片頂部使用同一個水平空間承載左側路線/站點預覽與右側候車資訊。上一輪布局優先嘗試讓站點預覽使用更多可用空間，但真實長站名如 `興華邨豐興樓 → 海底隧道巴士轉乘站` 會讓左側文本接近或視覺侵入右側候車區，甚至因兩行展示而把頂部區域撐高。

本次設計的產品判斷是：結果列表是高頻掃視界面，卡片高度穩定和候車資訊穩定比完整展示長站點名更重要。站點預覽仍有價值，但它應被約束在左側固定可用區域內，超出時單行省略。

## Goals / Non-Goals

**Goals:**
- 保持現有卡片風格、右側候車資訊寬度、候車文案、下一班摘要和鈴鐺樣式不變。
- 將卡片頂部劃分為左側文本區與右側固定候車區，站點預覽不得侵入右側候車區。
- 站點預覽固定單行展示，超出左側可用寬度時使用尾部省略。
- 左側路線號 + 站點預覽的整體高度與右側候車區保持一致，不再因長站點名撐高頂部區域。
- 沒有下一班摘要或大字體模式下，仍保留右側候車區完整寬度，避免卡片間布局跳動。

**Non-Goals:**
- 不改變站點預覽文案生成規則、資料來源或 Citybus API 解析。
- 不改變候車 ETA、下一班 ETA、通知欄監控或語音播報邏輯。
- 不追求長站點名完整展示。
- 不縮小右側候車區寬度、不縮小鈴鐺觸控區、不改變鈴鐺透明低權重樣式。
- 不改變卡片底部價格、耗時、步行距離展示。

## Decisions

### 1. 右側候車區作為固定保留區

卡片頂部 SHALL 將右側候車區視為不可被左側文字侵入的固定保留區。實作可以繼續使用 `FrameLayout` + end margin，也可以改為 `ConstraintLayout`，但行為上需要滿足：

```text
左側文本區                         固定右側候車區
┌──────────────────────────┬────────────────────┐
│ 118                      │ 等候 3 分鐘   鈴鐺 │
│ 興華邨豐興樓 → 海底...    │ 下一班 20 分鐘 ›   │
└──────────────────────────┴────────────────────┘
```

右側保留區寬度應覆蓋主候車、鈴鐺和下一班摘要。即使下一班摘要不可見，也不應把站點預覽臨時放寬到右側區域，因為這會造成不同卡片寬度和省略點不一致。

### 2. 站點預覽強制單行省略

站點預覽 SHALL 使用 `maxLines=1`、`ellipsize=end` 或等效策略。它不應再使用兩行降級，也不應因長站名把「乘站」等尾部文字換到下一行。

這是刻意取捨：長站名完整性讓位於卡片高度穩定。用戶若需要完整站點細節，可通過卡片詳情或後續詳情入口查看，而結果列表保持緊湊掃視。

### 3. 左右頂部區域同高

頂部信息區應以右側候車區高度作為穩定高度基準。左側路線號和站點預覽應在同一高度內垂直排布：

- 路線號位於左側上方。
- 站點預覽位於路線號下方。
- 左側整體不因站點文案變長而增高。
- 分隔線與底部價格/耗時/步行行的位置不因站點文案變長而下移。

### 4. 大字體與無下一班策略

大字體模式下可以維持既有的下一班摘要隱藏策略，但站點預覽仍不得因此侵入右側候車區，也不得改為兩行。無下一班摘要時也保留完整右側候車區寬度，確保所有卡片左側文本區寬度一致。

## Risks / Trade-offs

- [Risk] 長站點名會更早省略，部分終點名稱不可完整讀取。→ Mitigation：這是本次明確選擇；卡片詳情可承載完整站點信息，列表優先保持高度穩定。
- [Risk] 固定右側保留區會讓短候車文案時左側可用寬度看起來偏保守。→ Mitigation：保持卡片間省略點一致，降低掃視成本。
- [Risk] 大字體下單行省略可能讓站點預覽更短。→ Mitigation：大字體場景優先避免重疊和卡片高度膨脹。
- [Risk] 改動布局約束可能影響候車區點擊或鈴鐺點擊。→ Mitigation：保留既有 view id 和點擊事件 contract，補充 UI tree 或 contract 驗證。
