## Context

当前 `citybus-route-query` 已经把主界面查询切到 Citybus `ppsearch_p3.php`，并解析 `routelist2` 中的候选巴士路线。用户进一步验证发现，`m1` 参数的 `T`、`F`、`W` 三个枚举值会返回不同路线集合；单独使用 `m1=T` 会遗漏部分多程路线。

Citybus 返回 HTML 中还存在两个当前模型未覆盖的信息：

- `步行距離(約) XXX米`：需要提取为步行距离分钟以外的独立数值列。
- `免費` / `免費 *`：需要按 0 HKD 计入总价格，而不是因为不是 `$x.x` 或 `港元x.x` 格式而解析失败。

本次变更仍基于 Kotlin + XML、`HttpURLConnection`、jsoup、Repository 分层，不引入新的网络框架或 UI 框架。

## Goals / Non-Goals

**Goals:**

- 对同一条已保存路线并发查询 `m1=T`、`m1=F`、`m1=W` 三种 Citybus 结果。
- 三次请求共用同一个香港时间 `t`。
- 保持当前 App 的其他 query 参数不变，不新增 `ws`、`loc`、`ssid`、`sysid`。
- 将三次返回的候选路线聚合为一个结果列表。
- 使用 `路线号序列 + 总价格 + 总耗时 + 步行距离` 作为完全一致去重键。
- 默认按总耗时由低到高排序，并让 UI 表头展示 `总耗时(分钟) ↑`。
- 解析步行距离并新增 `步行距离(米)` 列。
- 支持 `免費` / `免費 *` 价格，按 0.0 HKD 计入总价。
- 表格支持横向滚动并支持五列表头排序。
- 使用 `漁灣邨漁進樓 -> 港鐵中環站` 在模拟器中验证多程路线、三种 `m1` 返回差异、步行距离和免费价格。

**Non-Goals:**

- 不新增 `ws`、`loc`、`ssid`、`sysid` 参数。
- 不实现 Citybus 会话初始化或动态 Cookie 管理。
- 不改变预计汽车到站时间的阶段性规则；仍暂时等于总耗时。
- 不新增路线详情页、地图展示或步行路线详情。
- 不把 `m1` 模式暴露为用户可选控件。

## Decisions

### 决策 1：在仓库层并发请求三种 `m1`

`CitybusBusRouteRepository.searchRoutes()` 负责一次用户查询的完整业务语义，因此三种 `m1` 查询应在仓库层完成，而不是让 `MainActivity` 管理三次请求。

实现方向：

- 新增 `CitybusRouteSearchMode` 或使用内部列表保存 `T/F/W`。
- 点击查询时生成一次 `queryTime`。
- 使用后台线程池并发发起三次 HTTP 请求。
- 每个请求只替换 `m1` 参数，其余参数保持当前实现。
- 每个 HTTP 响应由 `CitybusRouteParser` 解析为 `List<BusRouteOption>`。

理由：

- UI 只关心“查询某个起终点的路线结果”，不需要知道 Citybus 的多模式细节。
- 仓库层更适合统一处理部分失败、聚合、去重和默认排序。
- 方便单元测试 URL 构造和聚合逻辑。

备选方案：

- 在 `MainActivity` 中循环调用仓库三次：会把外部 API 细节泄漏到 UI。
- 串行请求三种模式：实现简单但查询耗时会明显增加。

### 决策 2：部分成功即可展示

三次请求中只要至少一次成功，就展示成功请求解析出的聚合结果；只有三次都失败才视为查询失败。

理由：

- `m1` 的三种结果是互补集合，部分可用比全部失败更有价值。
- Citybus 某个模式短暂失败时，用户仍可看到其他可用路线。

备选方案：

- 任意一次失败则整体失败：一致性强，但对外部接口波动过于敏感。

### 决策 3：使用结构化去重键

新增内部去重键，包含：

- 路线号序列：例如 `["8X", "10"]`，而不是只使用展示字符串。
- 总价格：免费段按 0 累计后的总价。
- 总耗时。
- 步行距离。

只有四者完全一致才认为是重复结果。

理由：

- 用户已明确要求完全一致才去重。
- 同一路线号可能在不同模式下有不同耗时或步行距离，不能合并。

备选方案：

- 只按路线号去重：会丢掉不同模式下的有效差异。
- 按完整 HTML 去重：过于脆弱，且不利于测试。

### 决策 4：扩展 `BusRouteOption` 保存步行距离和路线序列

将模型扩展为更适合聚合和排序的结构：

```kotlin
data class BusRouteOption(
    val routeName: String,
    val routeSegments: List<String>,
    val priceHkd: Double,
    val durationMinutes: Int,
    val arrivalMinutes: Int,
    val transferCount: Int,
    val walkingDistanceMeters: Int
)
```

`routeName` 仍由 `routeSegments.joinToString(" → ")` 生成。保留 `routeSegments` 的原因是去重需要“路线号序列”，不应依赖 UI 展示字符串。

如果为了降低改动范围，也可以不公开 `routeSegments`，改为在 parser 内部构造去重 key。但保留字段更利于排序测试和后续路线详情扩展。

### 决策 5：parser 同时支持 `港元`、`$` 和 `免費`

解析优先级保持现状：

1. 优先解析 `aria-label`。
2. `aria-label` 不完整时使用 table 文本兜底。

价格解析规则：

- `路线号 港元8.1` 解析为该路线段价格 8.1。
- table 文本中的 `$8.1` 解析为 8.1。
- `路线号 免費`、`免費 *` 或 table 文本中的 `免費 *` 解析为 0.0。

步行距离解析规则：

- 从 `步行距離(約)450米` 或 `步行距離(約) 450米` 提取 450。
- 如果候选路线缺少步行距离，则该候选视为无法完整解析，不进入结果列表。

理由：

- 用户新增列要求步行距离必须可用。
- 免费价格是 Citybus 正常返回格式，不能导致候选路线被过滤。

### 决策 6：默认排序由仓库或主界面统一应用

聚合后的默认排序必须是总耗时升序，并且 UI 表头显示当前排序状态。实现上有两种可接受方式：

- 仓库返回前按总耗时升序，`MainActivity` 设置 `sortField=DURATION`、`sortDirection=ASC`。
- 仓库只返回去重结果，`MainActivity` 调用 `BusRouteSorter.sort(..., DURATION, ASC)`。

推荐第二种，因为排序状态属于 UI 表格行为，主界面更容易同步表头箭头。

### 决策 7：五列表格使用横向滚动

结果区域外层增加水平滚动容器，表头和行使用一致的固定最小宽度，使五列在小屏上不挤压变形。

实现方向：

- 表头和 RecyclerView 放入同一个水平滚动区域，或表头/列表分别放入同步宽度容器。
- 结果行宽度与表头宽度一致。
- 列顺序：
  1. `路线`
  2. `价格(HKD)`
  3. `总耗时(分钟)`
  4. `预计汽车到站时间(分钟)`
  5. `步行距离(米)`
- 步行距离单元格只显示数值。

理由：

- 用户明确允许横向滚动。
- 五列固定同屏会牺牲路线名和长表头可读性。

## Risks / Trade-offs

- [Risk] 三次并发请求增加外部 API 压力和总流量。→ Mitigation：只在用户点击查询时触发，且每次只有 3 个 GET；共用同一个时间戳，避免重复无意义请求。
- [Risk] 部分请求失败可能导致结果不完整但用户无感知。→ Mitigation：第一版按用户要求至少一次成功即展示；后续可增加轻量提示或调试日志。
- [Risk] Citybus HTML 中 `免費` 或步行距离格式继续变化。→ Mitigation：parser 同时覆盖 aria-label 和 table 文本，并用用户样例新增单元测试。
- [Risk] 横向滚动表头和列表可能宽度不同步。→ Mitigation：使用统一行宽和列宽常量，模拟器截图/ADB UI 树验证。
- [Risk] `routeSegments` 扩展模型影响现有 Mock 和排序测试。→ Mitigation：同步更新 Mock、adapter 和所有模型构造测试。

## Migration Plan

- 扩展数据模型和排序字段。
- 扩展 parser，增加免费价格和步行距离解析。
- 扩展仓库，支持 `m1=T/F/W` 并发查询、部分成功聚合、去重。
- 更新主界面默认排序状态和五列表格 UI。
- 更新 Mock 数据和单元测试。
- 运行 `./gradlew testDebugUnitTest` 和 `./gradlew build`。
- 在 Pixel 8 模拟器中用 `漁灣邨漁進樓 -> 港鐵中環站` 验证真实查询结果。
