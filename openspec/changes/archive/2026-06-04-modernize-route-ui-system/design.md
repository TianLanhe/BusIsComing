## Context

当前 BusIsComming 已经完成主页面路线选择、路线查询、ETA 渐进补全、路线管理、克隆路线、重复路线校验和地点搜索加载反馈。现有 UI 仍以表格为主：`activity_main.xml` 中结果区域包含五列表头，`item_bus_route.xml` 按列权重展示结果，排序交互绑定在表头点击上。路线管理页已有卡片雏形，但页面整体仍偏基础列表；路线编辑页的起点/终点交换是整行文字按钮。

用户已经确认后续主风格采用 `docs/ui-style-guide.md` 中的“安静实用的现代通勤工具”，结果区域改为卡片，路线管理页也随主页面改造；本轮不需要引入 Compose。实现应继续遵守项目约定：XML + AppCompat + Material Components，保持 Repository 和现有数据模型边界不变。

## Goals / Non-Goals

**Goals:**

- 主页面结果从表格改为路线结果卡片，提升路线、候车时间和价格的阅读优先级。
- 主页面结果字号更舒服，不再为五列表格压缩到 11sp/12sp。
- 查询中、无结果、失败和 ETA 补全状态使用更现代、清楚的状态区域或状态卡。
- 排序入口从表头点击解耦，改成用户更容易发现的排序控件。
- 路线管理页使用同一视觉语言，保留新增、编辑、克隆、删除和空状态入口。
- 路线编辑页将交换起点/终点改为弯曲双向箭头图标按钮，保持原有交换逻辑、地点选择校验和重复路线校验。
- 将 `docs/ui-style-guide.md` 作为后续 UI OpenSpec 的默认参考。

**Non-Goals:**

- 不改变 Citybus 查询 API、ETA 补全策略、地点搜索 API 或 SQLite 数据结构。
- 不引入 Compose 或新的 UI 框架。
- 不新增地图、地址联想以外的地点能力，也不改变下拉候选列表的业务规则。
- 不重新设计 App 导航结构；本次仍保留主页面、路线管理页和路线编辑页的 Activity 结构。

## Decisions

### 1. 结果列表使用 RecyclerView 卡片替代表格

主页面保留 `RecyclerView`，但将 `item_bus_route.xml` 从横向五列改为卡片布局。卡片首行突出路线号、候车时间和价格，次行展示总耗时和步行距离。这样能把核心信息放大到 18sp 到 24sp，同时避免表格列宽压缩造成的换行和横向滚动。

替代方案是继续调大五列表格字号，但在常见手机竖屏上会让表头和单元格继续抢空间；另一种方案是引入 Compose 重写列表，但本轮只是视觉和交互重排，XML + RecyclerView 足够直接。

### 2. 排序控件从表头点击迁移到独立区域

排序仍使用现有 `SortField`、`SortDirection` 和 `BusRouteSorter`，但 UI 上改为结果区顶部的 chips、分段控件或紧凑按钮组。当前排序项和方向需要明确展示，点击同一排序项切换升降序，点击不同排序项默认升序。

这样保留现有排序行为和测试价值，同时避免用户必须猜测表头可点。表头被移除后，排序控件也不会和卡片布局耦合。

### 3. 状态反馈使用统一的状态卡/状态区域

`emptyResultText` 当前承担查询中、无结果和失败三种状态，视觉上不够稳定。本次应抽象为可复用的状态容器或至少统一样式：查询中显示非阻塞进度和文案，无结果提供清楚提示，失败提供错误文案。ETA 渐进补全仍在卡片中的候车时间位置显示 `...` 或 `-`，不阻塞列表滚动或排序。

### 4. 路线管理页跟随主页面风格

`activity_route_manage.xml` 继续使用列表结构，`item_route_config.xml` 继续作为路线卡片，但需要统一页面标题、按钮层级、卡片间距、空状态和卡片内操作样式。新增路线是页面主操作；编辑、克隆、删除是卡片内次级操作，删除保留二次确认。

### 5. 起终点交换使用图标按钮

路线编辑页保留现有 `swapPlaces()` 逻辑，把 `swapPlacesButton` 从文字按钮替换为右侧中线附近的图标按钮。图标语义为弯曲双向箭头，点击区域不小于 48dp，`contentDescription` 为“交换起点和终点”。点击后使用 150ms 到 250ms 的轻量旋转或淡入反馈，不阻塞输入。

布局上建议把起点和终点输入框放入同一相对定位容器或 ConstraintLayout，让图标位于两个输入框右侧之间；若继续使用 LinearLayout，也必须保证图标不会压缩输入框文本或遮挡下拉箭头。

### 6. 风格沉淀以文档为准，OpenSpec 引用文档

`docs/ui-style-guide.md` 是后续 UI 设计默认来源。本 change 的实现应遵循该文档，不把所有视觉细节硬塞进 spec；spec 只约束可验证的用户行为和关键视觉结果，例如卡片、字号、状态、触控区域和一屏结果数量。

## Risks / Trade-offs

- 卡片变高导致一屏结果数量不足 -> 控制卡片垂直 padding、间距和字体层级，目标是一屏 5 到 6 条完整结果；若和可读字号冲突，优先可读性。
- 移除表头后用户找不到排序 -> 在结果区顶部放置明确排序控件，并显示当前排序方向。
- 图标按钮位置与 TextInputLayout 的下拉箭头冲突 -> 使用外部独立 48dp IconButton，不放入输入框 endIcon 位置覆盖 Material 下拉行为。
- 新样式和旧 drawable/color 混用导致视觉不统一 -> 优先补充少量共享资源，例如卡片背景、状态卡背景、排序 chip 颜色和交换图标，而不是在每个布局里硬编码。
- 没有 Compose 会限制复杂动效 -> 本轮只做 150ms 到 300ms 的基础 View 动画，避免扩大技术栈。

## Migration Plan

1. 先新增或调整共享颜色、drawable、尺寸和图标资源。
2. 改造主页面结果区域：移除表头依赖，新增排序控件和状态区域，替换结果 item 为卡片。
3. 更新 `MainActivity` 的排序绑定、状态显示和结果刷新逻辑。
4. 改造路线管理页和路线卡片视觉，保持已有编辑、克隆、删除回调。
5. 改造路线编辑页交换控件布局和动画，复用现有 `swapPlaces()`。
6. 通过构建和模拟器/真机截图验证主页面、结果列表、管理页和编辑页没有文字重叠、不可点按钮或布局跳动。

## Open Questions

无。用户已确认结果改为卡片，主风格以 `docs/ui-style-guide.md` 中的第二种“安静实用的现代通勤工具”为主，局部吸收动效和状态卡；本轮不引入 Compose。
