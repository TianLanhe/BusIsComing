## Context

当前 App 使用 Kotlin、XML、AppCompat 和 Material Components 构建界面。主界面结果表格的表头和结果行都依赖固定 `760dp` 宽度，导致未查询、查询后以及不同屏幕尺寸下的宽度表现不稳定；路线选择框目前把路线名称、起点和终点拼成一段文本，长地址可读性较差。路线编辑页已经支持异步地点搜索和 debounce，但搜索过程中没有明显加载反馈，用户难以判断输入是否触发匹配。

路线配置仍使用本地 SQLite 持久化，起点和终点必须来自 Citybus 地点候选。此次优化只调整本地 UI、交互和保存校验，不改变 Citybus 查询接口、地点搜索接口或数据库结构。

## Goals / Non-Goals

**Goals:**
- 让主界面结果表格在常见手机屏幕上与可用宽度一致，表头和结果行同宽，并保持表头文案不变。
- 优化已保存路线选择、路线管理、克隆、交换起终点和重复路线校验，让常用路线维护更高效。
- 给起点和终点搜索提供独立、非阻塞的加载动画和状态文案。
- 保持现有 XML + Material Components 技术栈，不引入新的外部依赖。

**Non-Goals:**
- 不重新设计整套视觉风格，不引入 Compose。
- 不改变 Citybus HTTP API、请求参数或响应解析规则。
- 不修改路线配置数据库字段和迁移策略。
- 不实现地图选点、地址联想以外的新地点选择能力。

## Decisions

1. 主界面结果表格取消固定 `760dp` 宽度，改为占满父容器可用宽度。
   - 结果表头和结果行使用相同列权重，外层不再依赖横向滚动承载固定宽度内容。
   - 行高使用 `minHeight` + `wrap_content`，字段设置合理 `maxLines` 和字号自适应，确保常见屏幕下一页约显示 5 到 7 行。
   - 表头文案保持现状，例如 `价格\n(HKD)` 和 `预计汽车到站时间\n(分钟)` 不做缩写。

2. 路线选择框仅在折叠态使用主标题/副标题展示。
   - 折叠态展示两行：第一行为路线名称，第二行为 `起点 -> 终点`。
   - 下拉列表仍沿用当前单项展示方式，避免扩大此次变更范围。
   - 实现上可继续使用 `MaterialAutoCompleteTextView`，通过两行文本或富文本样式区分主副标题。

3. 路线克隆通过新增页承载，不直接写入数据库。
   - 路线管理页点击“克隆”后打开路线编辑页的新增模式。
   - 编辑页预填名称、起点和终点，名称默认追加 `（副本）`，降低保存时与原路线完全重复的概率。
   - 用户保存前仍可修改任一字段。

4. 重复路线校验在保存流程中完成。
   - 三要素指 trim 后的路线名称、起点地点对象和终点地点对象。
   - 新增或克隆时，如果已有路线三要素完全一致，系统阻止保存。
   - 编辑时，当前路线自身不参与重复比较；但编辑成另一条已有路线的三要素时必须阻止保存。
   - Repository 增加面向校验的查询或判断方法，UI 不直接扫描 SQLite。

5. 起点/终点交换保留地点选择语义。
   - 两边都已有选中地点时，交换 `Place` 对象和显示文本。
   - 若某一边只有未确认的输入文本，则交换显示文本后清空对应选中地点状态，保存前仍要求重新从候选中选择。
   - 交换操作不触发保存，也不阻塞继续输入。

6. 地点搜索加载状态按字段独立管理。
   - 起点和终点分别显示小型 indeterminate 加载动画与 `正在匹配地点...` 文案。
   - debounce 等待阶段和后台搜索阶段都可以展示搜索进行中状态，但用户继续输入时旧搜索必须取消或过期。
   - 加载反馈不能禁用输入框，不能阻塞键盘输入。

## Risks / Trade-offs

- [Risk] 表头文案保持不变会让窄屏列宽更紧张。→ 通过统一列权重、允许换行、限制 maxLines 和字号自适应缓解。
- [Risk] `MaterialAutoCompleteTextView` 的折叠态两行展示可能与下拉列表复用同一文本源。→ 下拉列表 adapter 继续使用当前 label，折叠态单独设置选中项显示文本。
- [Risk] 克隆默认追加 `（副本）` 后仍可能与已有副本重复。→ 保存前重复校验作为最终约束。
- [Risk] 搜索 loading 动画和下拉箭头可能在输入框尾部竞争空间。→ 优先在输入框下方或旁侧展示独立加载区域，避免覆盖输入和下拉控件。
