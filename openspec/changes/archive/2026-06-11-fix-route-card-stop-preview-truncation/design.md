## Context

當前 `item_bus_route.xml` 使用 `FrameLayout`，左側 `busRouteTextColumn` 以 `layout_marginEnd="180dp"` 為右側候車資訊塊預留固定空間。這保住了 `等候 5 分鐘`、鈴鐺和下一班摘要，但也讓第二行站點預覽只能使用扣除整個候車塊後的寬度。截圖中的 `興華邨興翠樓 → 漁...` 正是這個問題：第二行右側仍有視覺空間，但文字列因固定 margin 已提前結束。

站點預覽與路線號、候車主標題、下一班摘要的信息優先級不同。它不應被候車主標題的固定寬度無條件壓縮；同時修復也不能讓候車時間、下一班摘要或鈴鐺觸控區重新變窄。

## Goals / Non-Goals

**Goals:**
- 正常手機寬度與正常字體下，完整展示常見長度站點預覽，例如 `興華邨興翠樓 → 漁灣邨`。
- 保持右側候車主標題可完整展示兩位數分鐘文案。
- 保持下一班摘要可完整展示兩位數分鐘文案與箭頭。
- 保持通知鈴鐺不小於 `48dp` 的觸控區和既有低權重視覺。
- 修復不依賴橫向滾動，不引入表格式布局。

**Non-Goals:**
- 不改變站點預覽資料來源、補全流程或 Citybus API 解析。
- 不改變候車 ETA、下一班 ETA、通知欄監控或語音播報邏輯。
- 不要求任意超長站名永遠單行完整展示。
- 不縮小核心字號來硬塞所有文字。

## Decisions

### 1. 從固定右邊距改為內容感知約束

實作應避免讓站點預覽整行固定扣除 `busWaitArea` 的完整寬度。建議將卡片頂部改成 `ConstraintLayout` 或等效約束：

- 路線號位於首行左側。
- 候車主標題和鈴鐺位於首行右側，鈴鐺保持 `48dp`。
- 下一班摘要位於右側第二行或候車區內，保持完整兩位數展示。
- 站點預覽位於路線號下方，右邊界優先約束到下一班摘要左側；當下一班摘要不可見時可延伸至卡片右邊界。

這比單純把 `180dp` 改成較小數值穩定，因為它按第二行真正存在的右側內容分配空間，而不是用首行候車區的最大寬度去壓縮站點預覽。

### 2. 常見站點完整展示，極端站名才降級

正常寬度下，以下文案應完整展示：

```text
興華邨興翠樓 → 漁灣邨
```

若上下車站名更長，優先策略為：

- 允許站點預覽在卡片內使用最多兩行，保持起點與終點語義完整。
- 若仍不足，再對中間部分省略，但不得只保留起點而讓終點不可辨識。
- 不接受常見站點在仍有空間時被省略成 `漁...`。

### 3. 不用犧牲候車區修復站點預覽

此前 polish change 要求候車主標題與下一班摘要完整展示，這仍然成立。本變更不應通過縮小鈴鐺、壓縮候車主標題、截斷下一班摘要來換取站點預覽空間。若空間緊張，應優先改布局分行或約束，而不是讓任一核心文本互相擠壓。

### 4. 測試用固定回歸文案鎖定問題

應新增或更新 layout contract 測試，至少檢查：

- XML 不再使用會讓站點預覽固定扣除整個候車區的 `layout_marginEnd="180dp"` 策略。
- 站點預覽支持更寬的可用區域、內容感知約束或最多兩行展示。
- `興華邨興翠樓 → 漁灣邨` 被列為視覺或 contract 回歸案例。
- 鈴鐺仍為 `48dp`，候車與下一班文字仍有完整展示約束。

## Risks / Trade-offs

- [Risk] 站點預覽改為兩行會讓單張卡片略高，首屏卡片數可能減少。→ Mitigation：只在常見單行無法完整展示時才使用兩行；正常短站名仍保持緊湊。
- [Risk] ConstraintLayout 重排可能影響候車區點擊或鈴鐺點擊。→ Mitigation：保留既有 view id，補充點擊事件 contract 測試。
- [Risk] 下一班摘要與站點預覽同在第二行時仍可能競爭空間。→ Mitigation：使用內容感知約束；極端情況下站點預覽可換行，下一班摘要仍保持兩位數完整展示。
- [Risk] 單純依賴截圖驗證容易漏掉不同屏寬。→ Mitigation：同時增加 XML/layout contract 測試和模擬器截圖驗證。
