## Context

当前主頁在沒有任何常用路線時只顯示一個簡單空狀態卡，文案偏「缺資料」，按鈕也只提供 `新增路線` 與 `查詢臨時起點和終點`。這對第一次安裝 App 的用戶不夠友好，也沒有展示保存常用路線後能看到的候車、車費、耗時和步行距離價值。

相關現況：

- 主頁 XML 仍使用 `activity_main.xml`，無路線時 `emptyRouteState` 顯示，`queryControls` 和 `resultSection` 隱藏。
- 正常主頁頂部已移除 `巴士查詢` 文案，保留 `乘車碼` 與 `管理路線` 兩個主要按鈕。
- 真實結果卡片由 `item_bus_route.xml`、`BusRouteAdapter.BusRouteViewHolder.bind()` 和 `RouteResultCardFormatter` 共同定義。
- 管理頁與編輯頁仍是 XML + Material Components，適合做小範圍視覺配套，不需要引入 Compose。
- `docs/ui-style-guide.md` 定義 App 風格為「安靜實用的現代通勤工具」；本 change 允許根背景使用克制的淡綠漸變，但仍避免裝飾球、厚重 hero 或營銷頁語氣。

## Goals / Non-Goals

**Goals:**

- 改善新安裝和刪空路線後的主頁第一印象，讓用戶一眼知道第一步是新增常用路線。
- 保留臨時查詢作為次級入口，文案改為 `直接查詢一次`，降低理解成本。
- 用一張 `示例預覽` 結果卡展示保存常用路線後的核心價值。
- 讓示例卡與真實結果卡片保持同源渲染，避免後續結果卡片改版時示例失真。
- 統一主頁、管理頁、新增／編輯／複製頁的根背景，形成更一致、更精緻的視覺基底。
- 在管理頁和編輯頁做輕微配套優化，讓新背景不顯得突兀。
- 增加克制的首次頁進入動效，提升首次感受但不干擾高頻使用。

**Non-Goals:**

- 不重做主頁已有路線後的正常查詢結構。
- 不改有路線時 `乘車碼` 和 `管理路線` 的位置與主要按鈕樣式。
- 不改 Citybus 查詢、ETA、排序、最近路線選擇、保存資料格式或 SQLite schema。
- 不改乘車碼拉起策略。
- 不重做管理頁或編輯頁流程；不新增欄位、不改搜尋候選行為、不改刪除確認。
- 不引入 Compose、動畫庫或新外部依賴。

## Decisions

### Decision 1: 無路線主頁改為首次引導，不做完整 onboarding

做法：當沒有保存路線且沒有臨時查詢上下文或結果時，主頁顯示首次引導版面。版面接近 `final-first-home-direction.html` 的方向：左上保留 tonal `乘車碼`，主體在屏幕中上偏中位置，順序為兩行主標題、純文字 `示例預覽`、示例卡、居中的既有窄寬按鈕組。主標題固定為兩行：`把常走的路線放在這裡，` 與 `出門前一按即查。`，不使用 eyebrow。主 CTA 為 `新增常用路線`，次 CTA 為 `直接查詢一次`。

原因：用戶要立即知道第一步；完整 onboarding 會增加一次性內容並偏營銷化，不符合高頻工具定位。無 eyebrow 能讓視覺更乾淨，文案也適用於「曾經用過但刪空路線」的狀態。

替代方案：保留原空狀態卡。否決原因是第一印象弱，仍像缺資料提示。替代方案：做多步 onboarding。否決原因是超出本期範圍且會拖慢首次查詢。

### Decision 2: 無路線時隱藏 `管理路線`，有路線後保持現狀

做法：無保存路線且顯示首次引導時，頂部左側只保留 tonal 樣式的 `乘車碼` 小入口；不顯示 `管理路線`。當存在保存路線或主頁處於正常查詢狀態時，頂部仍按現狀顯示 `乘車碼` 與 `管理路線` 兩個主要按鈕，位置和樣式不變。

原因：沒有路線時沒有可管理對象，顯示 `管理路線` 會和 `新增常用路線` 主 CTA 產生語義衝突。有路線後，管理入口語義完整，且用戶已確認正常主頁頂部不改。

替代方案：始終顯示兩個按鈕。否決原因是首次頁決策成本高且層級混亂。替代方案：無路線時完全移除頂部入口。否決原因是乘車碼是獨立旁路能力，不依賴常用路線。

### Decision 3: 示例卡复用真實結果卡片 binder

做法：抽出可復用的結果卡片綁定器，例如 `BusRouteCardBinder` 或等效封裝。`BusRouteAdapter.BusRouteViewHolder` 和首次頁示例卡都使用這個 binder。示例資料使用本地固定 sample `BusRouteOption` 或等效 immutable UI model，不發起網路、不保存、不計入查詢結果。示例卡外層只加 `示例預覽` 標籤；卡片本身不可點擊，不打開詳情、ETA 班次或通知監控。

原因：用戶明確要求示例卡「尽可能接近真实卡片，而且要跟随真实卡片变化」。共用 binder 能讓後續卡片 layout/formatter 變更自然同步，避免維護兩套 UI。

替代方案：在空狀態 layout 中手寫一張靜態卡。否決原因是會與真實卡片逐步分叉。替代方案：直接嵌入一個 RecyclerView。否決原因是為單張不可交互示例過重，並且會引入不必要的列表生命週期。

### Decision 4: 臨時查詢結果優先於首次頁

做法：無保存路線時，如果用戶點 `直接查詢一次` 並成功發起臨時查詢，主頁隱藏首次引導，顯示既有臨時查詢上下文、查詢狀態和結果區。若只打開底部彈層後關閉且沒有發起查詢，仍顯示首次引導。保存臨時查詢為常用路線後進入正常有路線主頁。

原因：用戶剛完成一次查詢時，結果比首次引導更重要；不能讓首次頁覆蓋結果。

替代方案：無路線時始終顯示首次頁並把臨時結果放在下方。否決原因是第一屏會變擁擠且查詢結果不明顯。

### Decision 5: 全 App 完整頁根背景使用克制漸變

做法：新增或調整根背景 drawable，使用淡綠到近白的垂直漸變。主頁根容器、結果區背後、路線管理頁和路線編輯頁都使用同一背景。結果卡、管理卡、狀態卡、chips、底部彈層和對話框保持白色或淺色實體表面，不做玻璃化或半透明。新增／編輯／複製頁的主要表單功能區使用同色系、更克制的淡色漸變承載面，輸入框本身仍保持清晰的 outlined 實體控件。

原因：統一根背景能形成更精緻的一致性；保留白色內容表面可維持可讀性和工具感。

替代方案：只改首次頁背景。否決原因是首次頁與正常頁跳變明顯。替代方案：全局半透明／玻璃化。否決原因是資訊工具的穩定性和可讀性下降。

### Decision 6: 管理頁和編輯頁只做輕微配套

做法：在新背景下微調管理頁和編輯頁的標題層級、副文案、間距、表單承載面或列表留白。新增／編輯／複製頁保持接近原有的垂直居中位置，表單功能區使用淡色漸變，路線表單輸入、交換按鈕、保存按鈕和候選列表流程保持原有能力。

原因：只換背景會讓表單和標題直接浮在漸變上，與新主頁精緻感不一致；但重做管理/編輯頁會擴大 scope。

替代方案：只換背景不調整配套。否決原因是視覺落差較大。替代方案：完整重做管理和編輯頁。否決原因是超出首次主頁優化的核心目標。

### Decision 7: 首次頁進入動效限於一次性輕量反饋

做法：首次引導顯示時，標題、示例卡和按鈕依次做 150 到 250ms 淡入／輕微上移。動畫只針對首次引導容器，不在正常主頁和結果列表中新增額外動效。若系統動畫關閉或動畫 scale 為 0，UI 直接顯示最終狀態。

原因：首次頁是第一印象場景，克制進場動效能提升質感；限制範圍可避免高頻使用干擾。

替代方案：不做任何動效。否決原因是錯過提升第一印象的小成本機會。替代方案：循環裝飾動畫。否決原因是不符合工具定位且可能干擾操作。

## Risks / Trade-offs

- [示例卡被誤認為真實結果] → 使用 `示例預覽` 標籤，禁用卡片交互，不顯示在查詢結果 list 中，不持久化，不計入查詢狀態。
- [共用 binder 重構影響真實結果卡] → 先補充 binder/formatter 單元測試或 contract 測試，再遷移 ViewHolder；保持 `item_bus_route.xml` 作為單一 layout 來源。
- [漸變背景降低可讀性] → 主頁與管理頁文字仍放在白色卡片、輸入框或狀態表面上；新增／編輯／複製頁的功能區只使用低對比淡色漸變，輸入框仍保持 outlined 控件；驗證正常字體、窄屏和結果列表場景。
- [無路線與臨時查詢狀態切換出錯] → 明確引入或整理 MainActivity 顯示狀態判斷：首次頁、正常路線控制、臨時結果區互斥。
- [動畫導致 layout 跳動或測試不穩] → 動畫只改 alpha/translationY，不改 layout params；测试中可等待最終狀態，並保證動畫關閉時正常顯示。

## Migration Plan

1. 新增根背景 drawable 和必要色值，先將主頁、管理頁、編輯頁根容器切換到新背景。
2. 抽出結果卡片 binder，讓現有 RecyclerView 使用 binder；用測試確認真實結果卡文案和可點擊狀態不變。
3. 新增首次引導 layout 或重構 `emptyRouteState`，接入示例卡和新 CTA 文案。
4. 更新 MainActivity 狀態切換：無路線無臨時結果顯示首次頁；臨時查詢中/結果顯示結果區；有路線顯示正常主頁。
5. 微調管理頁與編輯頁視覺配套。
6. 補充/更新單元與 instrumentation 測試，並在模擬器截圖驗證無重疊、按鈕可辨識、結果列表可讀。
