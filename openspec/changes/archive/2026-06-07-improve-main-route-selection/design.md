## Context

主頁目前以單一已保存路線下拉框作為查詢入口，`MainActivity` 以 `selectedRoute: RouteConfig?` 作為唯一查詢來源。這讓高頻常用路線查詢足夠直接，但偶發的臨時起終點查詢必須先進入新增路線流程，操作成本偏高。

路線新增/編輯頁已具備 Citybus 地點搜尋、候選選中、修改文本後清空已選地點、起終點交換和保存校驗。主頁新增臨時查詢時應復用這些語義，避免出現一套更寬鬆的自由文本查詢。

同時，主頁仍應服務最高頻操作：用戶打開 App 後快速點選常用路線並查詢。因此臨時查詢不應常駐在主頁，不應增加分段切換，也不應在界面外露使用次數或排序規則。

## Goals / Non-Goals

**Goals:**
- 主頁只保留一個「常用路線」區塊，讓常用路線卡片本身承擔選擇與快捷切換。
- 常用路線不足 3 條時卡片自適應鋪滿可用寬度；超過 3 條時展示前 3 條與右側展開箭頭。
- 完整常用路線列表從右側箭頭進入，列表最後一項提供臨時查詢入口。
- 臨時查詢使用底部彈層，包含起點、終點、交換、查詢和保存為常用。
- 常用路線查詢時記錄使用次數與最近使用時間，並在主頁按使用統計排序。
- 地點選擇、校驗和保存仍遵循既有 RouteConfig/Place 模型。

**Non-Goals:**
- 不新增首頁常駐起點和終點輸入框。
- 不新增「常用路線 / 臨時查詢」分段切換。
- 不在 UI 展示使用次數、最近使用時間、排序規則或排名。
- 不改動 Citybus 路線查詢、ETA、結果解析、結果卡片詳情能力。
- 不在本次變更加入手動置頂、拖拽排序或常用路線分組。

## Decisions

### Decision: 使用單一「常用路線」區塊，而不是已選大卡加快捷小卡

主頁只展示「常用路線」區塊。區塊內的卡片同時代表可選路線和當前選中路線，當前選中卡片以高亮狀態標識。

原因：
- 避免同一條路線在首頁出現兩次，降低視覺重複。
- 常用路線卡片本身就是最直接的高頻入口。
- 主頁保留更多空間給查詢按鈕和結果列表。

替代方案：
- 保留已選路線大卡加 2-3 個快捷小卡。該方案資訊明確，但首頁重複展示路線選擇，與最終確認的輕量方向不符。
- 使用分段切換常用/臨時。該方案清楚但讓低頻臨時查詢成為首頁一級模式，增加主頁負擔。

### Decision: 超過 3 條常用路線時首頁只展示前 3 條與展開箭頭

首頁按排序後的前 3 條常用路線展示快捷卡片；更多路線通過右側單獨箭頭打開完整列表。少於 3 條時，卡片使用等寬或自適應寬度鋪滿可用寬度。

原因：
- 讓常見手機寬度下首頁保持穩定、可掃讀。
- 3 個快捷入口足以覆蓋高頻路線，同時避免橫向滾動影響效率。
- 單獨箭頭不佔用第三個路線卡位，語義比「更多」卡片更清晰。

### Decision: 臨時查詢入口放在完整常用路線列表最後一項

臨時查詢不出現在首頁默認狀態。用戶點擊常用路線區右側箭頭後，完整列表最後一項顯示 `查詢臨時起點和終點`。

原因：
- 臨時查詢是低頻能力，不應佔首頁默認空間。
- 它和「選擇查詢路線」語義相近，放在路線列表底部自然可發現。
- 放在最後一項能保持常用路線選擇的主優先級。

替代方案：
- 首頁查詢按鈕下方放文字入口。更容易發現，但會長期佔首頁空間。
- 首頁常駐兩個地點輸入框。臨時查詢最快，但會破壞常用路線查詢優先級。

### Decision: 臨時查詢使用底部彈層

點擊 `查詢臨時起點和終點` 後打開底部彈層。彈層包含起點與終點 autocomplete、交換圖示按鈕、`使用此路線查詢` 主按鈕與 `保存為常用` 次要入口。

原因：
- 底部彈層能完成低頻輸入任務，同時保留主頁上下文。
- 不需要跳轉頁面，查詢完成後可直接回到結果列表。
- 之後若保存為常用，可復用 RouteConfigRepository 和現有保存校驗。

### Decision: 抽取可復用地點輸入控制邏輯

將 RouteEditActivity 中的地點輸入控制邏輯抽取為可復用組件或 helper，例如 `PlaceInputController`，由新增/編輯頁和臨時查詢彈層共同使用。

原因：
- 地點搜尋有 debounce、序列號忽略舊結果、loading、候選 adapter、已選狀態、修改文本清空狀態等細節，直接複製容易分叉。
- 主頁臨時查詢和編輯頁必須保持同樣的有效地點語義。

### Decision: 使用次數排序只影響主頁查詢入口

路線表新增：
- `usage_count INTEGER NOT NULL DEFAULT 0`
- `last_used_at INTEGER`

主頁常用路線讀取順序：
1. `usage_count DESC`
2. `last_used_at DESC`
3. `updated_at DESC`
4. `id DESC`

用戶點擊常用路線執行 `查詢` 並確認起終點有效時，更新該路線 `usage_count + 1` 和 `last_used_at = now`。臨時查詢不計入任何路線；臨時查詢保存為常用後，後續從常用路線查詢時再累計。

原因：
- 使用次數反映真實高頻行為，能自動把常查路線排到前面。
- `last_used_at` 解決使用次數相同時的近期偏好。
- 不在 UI 展示排序原因，避免讓主頁變成數據看板。

## Migration Plan

- 將 `RouteConfigDbHelper.DATABASE_VERSION` 升級到新版本。
- 從目前地點結構資料庫升級時，應保留既有路線並為新增欄位填入預設值，而不是刪除現有路線。
- 從舊純文本資料庫升級到新版本時，仍可沿用既有策略刪除舊表並建立新表，因為舊資料缺少地點經緯度，無法可靠查詢。
- 若升級失敗，下一次啟動仍應能通過 SQLiteOpenHelper 的升級流程重試或重建表。

## Risks / Trade-offs

- [Risk] 新路線使用次數低，不容易出現在前三個快捷卡片。→ Mitigation: 完整列表仍可選擇全部路線；後續若真實使用需要，再加入手動置頂，不在本次先增加規則複雜度。
- [Risk] 臨時查詢入口藏在列表底部，部分用戶初次不容易發現。→ Mitigation: 列表最後一項使用清晰文案和輕量圖示，保持可發現但不搶常用路線優先級。
- [Risk] 地點輸入邏輯抽取會觸碰 RouteEditActivity。→ Mitigation: 先以行為等價為目標抽取，保留現有文案、debounce 和校驗規則，並用既有測試覆蓋回歸。
- [Risk] 查詢按下即計數可能把失敗查詢也計入使用。→ Mitigation: 計數代表用戶選擇使用該常用路線，不依賴 Citybus 網路狀態；失敗不應影響路線偏好統計。
