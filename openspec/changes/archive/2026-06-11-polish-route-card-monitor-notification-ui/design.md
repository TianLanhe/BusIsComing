## Context

當前結果卡片右側候車資訊塊固定寬度約 `138dp`，但同一行內還包含不小於 `48dp` 的通知鈴鐺觸控區，導致主候車文字實際可用寬度不足，`等候 NN 分鐘` 或 `下一班 NN 分鐘 ›` 在部分設備上會截斷單位。

通知欄監控目前的 notification content title 由服務端拼接為 `<routeName> 通知欄監控`，正文再串接狀態、候車、下一班、步行和更新時間。這種寫法信息完整但折疊態主次不清：用戶需要掃完整行才能判斷是否該出門。最新產品方向是先用標題展示監控狀態，再用正文展示距離最遲出門時間的倒計時與關鍵依據。

## Goals / Non-Goals

**Goals:**
- 常見手機寬度與正常字體下，候車主標題和下一班摘要完整展示兩位數分鐘文案。
- 保留通知鈴鐺不小於 `48dp` 的觸控區和低權重視覺樣式。
- 極窄屏或超大字體時，優先完整展示主候車狀態；下一班摘要可整行隱藏或省略。
- 通知內容標題統一為 `<首程巴士號> · <監控狀態>`。
- 折疊態正文按照「剩餘/已過、車到站、步行或下一班」排序。
- 展開態正文在折疊態基礎上補充下一班與更新時間。
- 通知 action 只保留 `刷新` 和 `停止`；通知本體點擊仍打開 App。
- 先使用 Android 標準通知模板和 `BigTextStyle`，不做 custom `RemoteViews`。
- 只修改 App 可控的 notification content title、content text、big text 和 action，不修改通知 channel 名稱或 Android 系統可能顯示的 App 名稱。

**Non-Goals:**
- 不重新設計結果卡片整體結構。
- 不縮小鈴鐺觸控區。
- 不要求三位數分鐘完整展示。
- 不改變刷新策略、語音播報或停止策略。
- 不新增 `刷新`、`停止` 以外的新 action；本變更會移除顯式 `打開 App` action。
- 不監控轉乘段 ETA；標題巴士號只使用首程巴士號。
- 不在本變更中引入 custom notification layout。

## Decisions

### 1. 候車資訊塊適度加寬，但仍保持穩定上限

實作應將右側候車資訊塊調整為能完整容納：

```text
等候 99 分鐘
下一班 99 分鐘 ›
```

鈴鐺仍保留 `48dp` 觸控區。候車資訊塊可採用固定寬度、最小寬度加最大寬度，或 ConstraintLayout percent/constraint width；重點是避免候車區無界擠壓左側路線與站點預覽。

建議以實機/模擬器截圖調整，不在 spec 中鎖死精確 dp；目前 `138dp` 不足，實作可先嘗試約 `168dp` 到 `176dp` 的區間，再依 UI 驗證微調。

### 2. 極端布局下優先保住主候車狀態

常見寬度下主副候車文字都應完整。若字體縮放或屏幕過窄造成空間不足：

- 主候車狀態優先完整展示，例如 `等候 12 分鐘`、`即將到站`、`暫無車輛`。
- 下一班摘要可以整行隱藏，或在保留語義的前提下省略。
- 不接受只顯示半截單位，例如 `等候 12 分...` 或 `下一班 28 分...`。

這比盲目壓縮字體更穩，符合現有「安靜實用」工具風格。

### 3. 通知使用狀態標題和倒計時正文

新增或調整 formatter，生成 notification content title：

```text
<firstLegRoute> · <monitorStatus>
```

規則：

- `firstLegRoute` 取被監控路線的首程巴士號，例如 `82X`。
- `monitorStatus` 取現有狀態顯示文案：`準備出門`、`立即出門`、`快遲到了`。
- 多段路線只取首程巴士號，不使用完整轉乘路線。
- 開始監控、刷新成功、刷新失敗、服務重建後恢復和 public notification 都應使用同一 title formatter。
- title 不再使用 `通知欄監控`，也不再在標題中展示常用/臨時路線上下文。

折疊態正文由同一 formatter 生成。正常狀態下：

```text
準備出門：剩餘 4 分鐘 · 車 7 分鐘到 · 步行 3 分鐘
立即出門：剩餘 1 分鐘 · 車 4 分鐘到 · 步行 3 分鐘
快遲到了：已過 1 分鐘 · 車 2 分鐘到 · 下一班 25 分鐘
```

`剩餘/已過` 的含義是相對最遲出門門檻的倒計時：

```text
waitMarginMinutes = firstEtaMinutes - walkingMinutes
```

- `waitMarginMinutes > 0`：展示 `剩餘 N 分鐘`。
- `waitMarginMinutes <= 0`：展示 `已過 N 分鐘`，其中 `N = abs(waitMarginMinutes)`；若剛好為 0，可展示 `已過 0 分鐘` 或按實作轉為 `立即出門` 下的 `剩餘 0 分鐘`，但應保持規則一致。

展開態正文使用標準 `NotificationCompat.BigTextStyle`，格式為：

```text
剩餘 2 分鐘 · 車 5 分鐘到 · 步行 3 分鐘 · 下一班 25 分鐘 · 12:45 更新
```

若進入 `快遲到了`：

```text
已過 1 分鐘 · 車 2 分鐘到 · 步行 3 分鐘 · 下一班 25 分鐘 · 12:45 更新
```

展開態保留步行時間，即使折疊態 `快遲到了` 以下一班替代步行。

### 4. 通知 action 只保留刷新和停止

標準通知模板保留 content intent，因此用戶點擊通知本體仍可打開 App。底部 action 只保留：

```text
[刷新] [停止]
```

理由：

- `打開 App` action 與通知本體點擊重複。
- `刷新` 和 `停止` 是通知層最直接的兩個操作。
- 暫不加入 `查看班次`、`靜音語音` 等新操作，避免本輪 polish 擴大交互範圍。

## Risks / Trade-offs

- [Risk] 候車資訊塊加寬會壓縮左側路線和站點預覽。→ Mitigation：只支持兩位數固定文案完整展示，不追求三位數；左側保持單行省略，右側設最大寬度。
- [Risk] 超大字體下主副候車都完整展示會讓卡片過高或擁擠。→ Mitigation：優先主候車狀態，必要時隱藏下一班摘要。
- [Risk] 標題不展示常用路線上下文後，用戶可能難以區分同一首程巴士的不同路線。→ Mitigation：本輪優先強化出門決策；若後續需要，可在展開態或 content text 補充上下文，而不搶占折疊態標題。
- [Risk] `剩餘` 可能被理解為車輛剩餘時間。→ Mitigation：正文緊接 `車 X 分鐘到` 和 `步行 M 分鐘`，並在設計/測試中明確 `剩餘` 表示距離最遲出門門檻。
- [Risk] Android 系統通知 UI 仍可能額外展示 App 名或 channel 名。→ Mitigation：本變更只承諾 App 可控的 notification content title。
