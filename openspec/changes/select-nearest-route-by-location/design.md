## Context

主頁目前透過 `RouteConfigRepository.getAll()` 取得已保存常用路線，並依使用次數、最近使用時間、更新時間和 id 排序。`MainActivity.loadRouteConfigs()` 在有常用路線時選中前一次選中的路線，否則選中排序後第一條路線；`RouteShortcutSelector.visibleRoutes(...)` 控制首頁最多 3 張快捷卡，並在選中路線不在原始 Top 3 時臨時提升到第一張卡。

每條 `RouteConfig` 已持久化起點與終點的 `Place(name, latitude, longitude)`。Citybus 地點搜尋回傳的每個 `Place` 同樣包含經緯度，因此主頁最近起點與候選距離都可以在本機用同一套直線距離計算，不需要資料庫遷移、Citybus 查詢或步行路線服務。

新增、編輯、複製路線及臨時查詢共用 `PlaceInputController`，選定地點後透過 `setSelectedPlace(place)` 把 `Place.name` 顯示在輸入框，並以 `Place.latitude`、`Place.longitude` 參與後續 Citybus 點到點查詢。`improve-route-refresh-feedback` 正在把候選結果修正為表單內嵌 `RecyclerView`，並將候選項調整為約 52dp 的白色圓角卡片。該 change 原本規定候選項只顯示地點名稱；本 change 在其完成後擴充同一候選項，在右側加入目前位置距離。

目前專案尚未宣告定位權限，也未加入 Google Play Services Location。通知權限請求已在 `MainActivity` 使用 `ActivityCompat.requestPermissions(...)` 與 `onRequestPermissionsResult(...)`；定位權限應沿用同一風格。定位取得、30 秒快取、3 秒超時和並發請求不應分散在 `MainActivity`、`RouteEditActivity` 與 `TemporaryRouteBottomSheet` 中。

## Goals / Non-Goals

**Goals:**

- 主頁打開時，在常用路線至少 2 條的情況下，自動嘗試一次定位並選中起點最近的常用路線。
- 接受精確或粗略定位，但在主頁自動選路時用位置精度與明顯領先規則降低誤選風險。
- 純新增路線與臨時查詢開啟時，在起點尚未預填且用戶尚未操作時，自動嘗試以目前位置作為起點。
- 所有路線表單與臨時查詢的起點輸入框提供內嵌定位按鈕，支援用戶手動改用目前位置；終點不提供該快捷入口。
- 以獨立 `PlaceNameResolver` 邊界把目前 GPS 位置轉為可顯示與可保存的 `Place.name`；本期先 mock 固定返回 `目前位置附近`。
- 在新增、編輯、複製路線及臨時查詢中，若 App 已有定位權限且位置可用，於候選項右側顯示目前位置到候選地點的直線距離。
- 候選搜尋與展示不等待定位；位置稍後返回時原位補充距離。
- 共用一次性定位、位置快照、直線距離計算與格式化能力，避免多套實作漂移。
- 定位失敗、未授權或能力不可用時穩定降級，不影響既有候選選擇與主頁常用排序。
- 保持完整常用列表排序、使用統計、Citybus 搜尋排序、查詢流程及資料格式不變。

**Non-Goals:**

- 不在候選列表中加入固定「我的位置」候選、定位 loading 行、錯誤行或重新取得位置行。
- 不在本期接入真實坐標轉地點名稱服務；mock resolver 只作為後續真實 resolver 的替身。
- 不為終點輸入框增加目前位置快捷按鈕。
- 不新增候選距離開關。
- 不提供步行距離、道路距離或預估步行時間。
- 不依距離重新排序或過濾 Citybus 候選。
- 不持續監聽位置，不建立前台定位服務。
- 不在選定後的輸入框中保留距離，也不把距離寫入 `Place` 或資料庫。
- 不修改 Citybus mobile、DATA.GOV.HK ETA 或任何巴士查詢 HTTP 參數。
- 不新增 `LocationManager` fallback。

## Decisions

### 1. 建立共用一次性定位協調元件

新增應用程序範圍可共用的定位協調元件，例如 `CurrentLocationCoordinator`。它封裝 `FusedLocationProviderClient`、最近位置檢查、目前請求共用、3 秒超時及結果模型，但不直接彈出權限視窗。

位置快照至少包含：

- `latitude`
- `longitude`
- `accuracyMeters`
- 可用於判斷新鮮度的 monotonic 時間

元件行為：

1. 呼叫端沒有粗略或精確前台定位權限時，立即回傳不可用結果，不請求權限。
2. 若記憶體快照或 Fused last location 在最近 30 秒內取得，立即回傳。
3. 否則以 `PRIORITY_HIGH_ACCURACY` 請求一次目前位置，最多等待 3 秒。
4. 同一時間的多個呼叫共用同一次底層定位請求。
5. 成功位置更新進程內快照；失敗不覆蓋仍有效的快照。

選擇原因：

- 主頁、路線編輯頁及臨時查詢需要相同的快取與超時語義。
- 共用元件可避免三個 UI 容器各自處理 Fused callback、超時與競態。
- 一次性請求符合頁面距離參考需求，不需要持續定位與額外耗電。

替代方案：

- 各頁面直接使用 `FusedLocationProviderClient`：改動直觀，但權限、快取、超時及測試會重複。
- 持續定位並以 observable 廣播：距離更新更即時，但超出需求，增加耗電與生命週期複雜度。
- 只使用 last known location：顯示快，但無法保證位置新鮮度或可用性。

### 2. 權限申請由明確定位價值入口觸發

App 宣告並請求 `ACCESS_FINE_LOCATION` 和 `ACCESS_COARSE_LOCATION`。權限請求不由「候選距離」本身觸發，而只由具備明確定位價值的入口觸發：

1. 主頁至少 2 條常用路線時，自動定位並嘗試選中最近起點。
2. 純新增路線與臨時查詢開啟時，若起點尚未由預填或用戶操作決定，自動嘗試以目前位置填入起點。
3. 用戶點擊起點輸入框右側的定位按鈕時，明確要求使用目前位置。

候選距離仍遵循輔助資訊原則：

- 若上述流程或既有權限已取得位置快照，候選距離可使用同一快照顯示。
- 若只是為了顯示候選距離而缺少權限，系統不彈權限、不顯示提示，候選仍純名稱展示。

自動定位權限拒絕狀態在主頁、純新增路線與臨時查詢之間共用：任一自動權限請求被拒絕後，後續自動入口不再彈系統權限視窗，避免多個入口輪流打擾用戶。用戶點擊起點定位按鈕屬於主動操作，可以再次觸發權限流程；若 Android 已不再顯示系統授權對話框，App 提供前往系統設定的恢復路徑。

若用戶之後在系統設定重新授權，所有使用共用定位元件的畫面自然恢復位置能力。若系統定位服務關閉，主動定位入口提示 `請開啟系統定位` 並提供設定入口；返回後自動重試一次，不進入循環重試。

選擇原因：

- 純新增路線與臨時查詢的起點高度可能就是目前位置，權限請求有直接任務價值。
- 編輯與複製路線已有既定起點，不應自動請求權限或覆蓋資料。
- 候選距離只是參考資訊，不應單獨打斷輸入流程。
- 手動定位按鈕給用戶可控入口，比候選列表固定「我的位置」行更清楚，也不污染 Citybus 候選結果。

替代方案：

- 仍只讓主頁請求權限：權限更保守，但新增路線和臨時查詢無法達成「起點預設目前位置」的目標。
- 在候選列表加入固定「我的位置」項：看似直觀，但會混入非 Citybus 候選、增加 loading／失敗狀態，並與候選列表原始排序語義衝突。
- 終點也加入定位按鈕：少數場景有用，但更容易誤觸，且終點通常不是目前位置，暫不納入。

### 3. 主頁每次建立後最多嘗試一次自動選路

`MainActivity` 建立後，如果常用路線數量至少 2 條，且本次 Activity 尚未嘗試自動定位，則啟動定位流程。`onResume()` 從管理頁返回時不重新定位。定位進行中先按既有常用排序顯示預設選中路線，不顯示額外 loading。

若定位結果返回前用戶手動選擇了路線，丟棄該結果，尊重用戶最新操作。自動選中不調用查詢、不增加使用次數、不更新最近使用時間。

替代方案：

- 每次 `onResume()` 都重新定位：會在返回主頁時覆蓋用戶選擇。
- 定位完成前不選中任何路線：破壞既有「打開即可查預設路線」體驗。

### 4. 主頁自動選路保留精度規則

新增純邏輯策略，例如 `NearbyRouteSelectionPolicy`，輸入位置快照與按使用統計排序後的 `RouteConfig` 列表，輸出是否可自動選中及 route id。

精度規則：

- `accuracy <= 500m`：直接選擇最近起點。
- `accuracy > 500m`：只有最近起點比第二近起點至少近 `accuracy` 米，才選擇最近起點。
- 其他情況不自動選中，回退常用排序並提示精度不足。

距離相同或非常接近時保留輸入列表順序作為 tie-breaker。

選擇原因：

- 主頁會自動改變目前路線，錯誤成本高於只提供參考資訊。
- 粗略位置仍可在明顯領先時生效，但避免香港密集起點下誤選。

### 5. 候選距離接受任何成功返回的位置

候選距離不套用主頁的 500 米精度門檻。只要共用定位元件成功返回位置，無論是粗略或精確權限、無論 `accuracyMeters` 大小，都計算並顯示距離，不增加「約」字樣。

選擇原因：

- 候選距離只供用戶參考，不會自動選擇或重新排序。
- 粗略距離仍能協助區分相距數公里的同名地點。
- 隱藏或添加不同文案會讓同一列表狀態反覆變化，增加理解成本。

替代方案：

- `accuracy <= 500m` 才顯示：更保守，但粗略位置權限下幾乎無法使用。
- 顯示 `約 1.2km`：語義更謹慎，但與已確認的緊湊格式不一致。

### 6. 抽出共用直線距離計算器

建立可 JVM 單測的共用地理距離計算器，例如 `GeoDistanceCalculator`，使用 Haversine 公式計算兩個 WGS84 經緯度座標的直線距離。

以下消費者使用同一計算來源：

- `NearbyRouteSelectionPolicy`
- 候選地點距離
- 現有通知監控中的 `WalkingTimeCalculator.straightLineDistanceMeters(...)`

位置取得與距離計算保持分離：定位元件只提供快照，計算器不依賴 Android `Location` 類別，便於 JVM 測試。

替代方案：

- 使用 Android `Location.distanceBetween(...)`：可靠但把純策略測試綁定 Android API。
- 保留各自 Haversine 實作：改動較少，但會形成重複公式與邊界差異。

### 7. 距離格式使用先取整、再決定單位

建立純格式化器，例如 `PlaceDistanceFormatter`：

1. 將非負距離四捨五入為整數米。
2. 取整後小於 1000 時顯示 `${meters}m`。
3. 取整後大於或等於 1000 時，以公里顯示一位小數，例如 `1.0km`、`1.2km`。
4. 使用固定數字格式，避免裝置語系把小數點或單位格式改成不一致形式。

因此：

- `368.4m` → `368m`
- `999.4m` → `999m`
- `999.6m` → `1.0km`
- `1249.6m` → `1.3km`

距離文字的可見格式不加空格、不加 `距你` 或 `約`。TalkBack 等無障礙描述使用完整語義，例如「距離目前位置 368 米」。

### 8. 候選項使用單行左右結構

在 `improve-route-refresh-feedback` 完成正式候選卡佈局後，將候選項擴充為：

```text
地點名稱                         [定位圖示] 368m
```

- 行高保持約 52dp。
- 地點名稱位於左側，16sp 主文字，單行尾端省略。
- 右側距離區域不收縮、不換行；使用 14dp 左右的綠色線框定位圖示及 13sp 次要文字。
- 地點名稱前不加圓點、地點圖示或其他裝飾。
- 整個候選行維持既有點擊與按壓反饋。
- 圖示本身視為裝飾；候選行提供包含地點名稱與完整距離語義的無障礙描述。

候選列表仍維持 Citybus 原始順序、最多 100 條，以及主驗收配置 4 至 6 個完整項目的高度目標。新增右側資訊不得增加行高或把候選改成雙行。

替代方案：

- 名稱下方顯示「距離目前位置」副標題：資訊更完整，但行高增加並減少可見候選數。
- 距離 chip：更突出，但視覺噪音偏高，容易被誤解為可篩選控件。
- 只顯示純距離文字：更簡潔，但用戶較難快速理解數字含義；已確認加入定位圖示。

### 9. 候選先展示，位置返回後原位更新

地點搜尋與定位相互獨立：

1. Citybus 候選返回後立即展示名稱。
2. 若位置已可用，首次綁定時同步顯示距離。
3. 若位置稍後返回，控制器只更新目前可見資料的距離 presentation，不重新提交候選資料、不重排、不關閉列表，也不改變 `RecyclerView` 滾動位置。
4. 無權限、定位失敗、超時或結果為空時，整個右側距離區域保持隱藏，不顯示 `--`、loading、helper、Toast 或錯誤卡。
5. 用戶選定候選後，輸入框只顯示 `Place.name`。

頁面或彈層關閉後，以生命週期 token／generation 忽略遲到位置結果。`RouteEditActivity` 每次建立最多啟動一次位置取得；每次臨時查詢彈層會話最多啟動一次，並優先復用 30 秒共用快照。

### 10. 目前位置起點使用獨立 resolver 邊界

新增 `PlaceNameResolver` 或等效邊界，負責把定位快照轉成可展示、可保存的 `Place.name`。本期先以 mock 實作固定返回 `目前位置附近`，不接入 Android `Geocoder`、香港政府 API 或其他 reverse geocoding 服務。成功後建立：

```kotlin
Place(
    name = "目前位置附近",
    latitude = currentLocation.latitude,
    longitude = currentLocation.longitude,
)
```

這個 `Place` 與 Citybus 候選選出的 `Place` 一樣進入 `PlaceInputController.setSelectedPlace(...)`，後續交換、校驗、查詢與保存流程都把它視為普通地點。Citybus 點到點查詢使用的是緯度與經度，因此本期 mock 名稱只影響輸入框、路線名稱提示與保存後展示，不影響巴士查詢座標。

入口行為：

- 純新增路線：頁面建立後，若起點尚未預填且用戶尚未編輯起點，啟動一次自動目前位置流程；成功後填入起點，且不自動聚焦終點、不彈鍵盤、不自動查詢。
- 臨時查詢：底部彈層打開後套用同樣自動目前位置流程；成功後只填入起點，仍由用戶手動選終點並查詢。
- 編輯與複製路線：保留既有起點，不自動請求權限、不自動覆蓋；若已有定位權限，仍可為候選距離預取位置。
- 所有路線表單與臨時查詢：起點輸入框右側提供定位按鈕，終點不提供。用戶點擊後代表明確要求改用目前位置。

UI 與競態：

- 定位按鈕放在 `TextInputLayout` end icon／等效內嵌位置，避免恢復之前輸入框變窄的問題；觸控目標至少 48dp，content description 為 `使用我的位置`。
- 定位進行中禁用重複點擊，但輸入框保持可編輯；若用戶在結果返回前輸入、清空或選擇其他起點，該次定位結果丟棄。
- 主動點擊按鈕時，成功後才替換既有起點；失敗、拒絕、超時或 resolver 失敗時保留原起點或原輸入。
- 自動流程失敗時，起點保持空白，顯示輕量 helper `暫時無法取得目前位置，請手動選擇起點`；主動點擊失敗時使用 Toast，避免覆蓋既有 helper。
- 定位階段最多等待 3 秒，名稱解析階段最多等待 3 秒，整體最多 5 秒；本期 mock resolver 應快速返回，但仍遵循同一抽象與過期結果丟棄規則。
- Activity 重建或 BottomSheet 重新建立時恢復既有輸入與狀態，不重複彈權限；仍符合條件且未被用戶操作取消的自動流程可復用進行中請求或 30 秒快照。

不採用候選列表固定「我的位置」項的原因：

- 候選列表應維持純 Citybus 地點結果，避免把 App 自造項與 Citybus 候選混排。
- 固定項需要處理 loading、失敗、重試、權限及排序語義，會讓列表狀態變複雜。
- 起點欄位內定位按鈕的意圖更明確，也不影響候選列表可見數量。

後續真實 resolver 可在不改 UI 與調用方的前提下替換 mock 實作，輸出建築／屋苑／道路附近等更精準名稱。

### 11. `附近` 是短暫主頁 UI 狀態

`MainActivity` 保存臨時 `nearbySelectedRouteId`。當目前選中路線由定位策略決定時，對應首頁快捷卡顯示 `附近`。若最近路線剛好是既有預設路線，仍顯示標籤。

用戶手動從快捷卡或完整列表選擇任意路線時清空標籤。若自動選中的路線不在原始 Top 3，沿用 `RouteShortcutSelector` 現有臨時提升規則。

## Risks / Trade-offs

- [Risk] `improve-route-refresh-feedback` 與本 change 都修改候選項佈局，且前者規定「只顯示名稱」。→ Mitigation: 明確規定實作順序，先完成並驗收正式候選卡，再由本 change 取代「只顯示名稱」限制並加入右側距離。
- [Risk] Google Play Services Location 不可用。→ Mitigation: 主頁回退常用排序；候選靜默省略距離，不維護第二套 `LocationManager` fallback。
- [Risk] 粗略位置顯示的候選距離可能不精準。→ Mitigation: 距離只作參考，不自動選擇或排序；主頁自動選路仍保留精度規則。
- [Risk] 定位 callback 晚於頁面關閉、彈層關閉或用戶手動選路。→ Mitigation: 使用生命週期／generation token 丟棄過期結果。
- [Risk] 多個畫面同時請求位置造成重複耗電。→ Mitigation: 共用 30 秒快照並合併同時進行的底層請求。
- [Risk] 候選距離更新造成列表閃爍或跳動。→ Mitigation: 不重新提交或排序資料，只更新距離 presentation，保持相同行高及滾動位置。
- [Risk] 右側距離壓縮長地點名稱。→ Mitigation: 距離區固定完整顯示，名稱使用單行省略；在窄屏與大字體下做 instrumentation 與模擬器驗證。
- [Risk] 純新增路線與臨時查詢現在會主動請求定位權限，可能與主頁自動請求形成重複打擾。→ Mitigation: 自動權限拒絕狀態在入口間共用；任一自動請求被拒絕後，其他自動入口不再彈窗，只有用戶點擊定位按鈕才再次進入授權流程。
- [Risk] 本期 mock 名稱 `目前位置附近` 不是真實地點名，保存後可讀性有限。→ Mitigation: 保留原始 GPS 座標並隔離 `PlaceNameResolver`；真實 reverse geocoding 在後續 change 替換，不影響本期 UI、查詢和保存流程驗證。

## Migration Plan

- 先完成並驗證 `improve-route-refresh-feedback` 的候選卡片、行高、4 至 6 項高度與巢狀滾動修正。
- 加入定位依賴與 manifest 權限，不需要資料庫遷移。
- 先建立共用位置快照、定位協調、距離計算、格式化能力及 mock `PlaceNameResolver`，再接入主頁最近路線、目前位置起點與候選距離。
- 在 `RouteEditActivity` 與 `TemporaryRouteBottomSheet` 中先接入起點定位按鈕，再接入純新增路線與臨時查詢的自動起點流程，最後把同一位置快照交給候選距離。
- 在正式候選卡上加入右側距離區域，依次接入路線編輯頁與臨時查詢。
- 若新定位流程失敗或不可用，主頁回退既有排序，候選回退純名稱顯示。
- 回滾時可移除定位接線與距離區域；既有路線資料、Citybus 搜尋及資料庫不受影響。
