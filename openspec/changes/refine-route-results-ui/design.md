## Context

主頁目前使用 XML + AppCompat + Material Components，結果區已由表格改為卡片列表，並已有排序 chips、結果摘要、狀態卡和臨時查詢上下文條。`docs/ui-style-guide.md` 要求主頁保持「安靜實用的現代通勤工具」風格，並讓常見手機豎屏首屏盡量展示 5 到 6 條完整結果。

現有 `activity_main.xml` 的主頁根容器使用 20dp 內距，標題列到查詢控件、查詢按鈕到結果區仍偏寬；`管理路線` 使用琥珀輔助色，而 `查詢` 使用主綠色。`MainActivity` 已保存 `currentQueryContext`，可以作為下拉刷新復用最近一次查詢的基礎。

臨時路線保存目前有兩個入口：結果區臨時上下文條的 `保存`，以及臨時查詢底部彈層內的 `保存為常用`。兩者都即時建立 `TextInputLayout + TextInputEditText` 作為 `AlertDialog` view。截圖顯示長預填名稱與輸入框 hint/底線出現視覺重疊，需要把保存對話框調整為更穩定的 Material 表單樣式。

## Goals / Non-Goals

**Goals:**

- 讓有結果的路線列表支持下拉刷新，重新查詢最近一次有效起終點。
- 保留當前排序字段與方向，刷新後仍按當前排序展示。
- 微幅壓縮主頁頂部到常用路線、查詢按鈕到排序/結果區的垂直留白。
- 保留舒適閱讀與觸控：按鈕高度、chips 高度和結果卡片內部資訊層級不為了塞更多內容而縮到擁擠。
- 將 `管理路線` 入口改為與 `查詢` 按鈕一致的主綠色背景。
- 修正臨時路線保存對話框中路線名稱輸入框的重疊問題，讓長名稱、hint、錯誤提示和操作按鈕都能清楚展示。

**Non-Goals:**

- 不新增自動定時刷新或背景輪詢。
- 不改動 Citybus HTTP API、ETA 解析、去重、排序算法或路線詳情面板。
- 不重做結果卡片字段、字號、摘要文案或排序字段集合。
- 不改變常用路線保存、編輯、刪除、克隆或完整列表行為。
- 不新增保存前的完整路線編輯頁跳轉；保存仍在當前對話框完成。

## Decisions

### Decision: 使用 SwipeRefreshLayout 承載結果列表刷新

在結果列表區加入 `androidx.swiperefreshlayout.widget.SwipeRefreshLayout`，包裹 `RecyclerView` 為主；排序 chips、結果摘要和臨時查詢上下文條保持在刷新容器外，避免下拉時控制區一起位移。刷新只在存在最近一次有效查詢上下文且當前未查詢時啟用。

替代方案：
- 只提供重新點擊 `查詢`。這不能滿足下拉刷新手勢，且對已滾動結果列表的用戶不夠順手。
- 包裹整個結果區。這會讓排序 chips、摘要和上下文條跟隨刷新手勢移動，視覺噪音較大。

### Decision: 刷新復用查詢快照，但不記錄常用路線使用統計

新增或抽取 `refreshCurrentQuery()`，從 `currentQueryContext` 取得最近一次 `Saved(routeId)` 或 `Temporary(origin, destination)` 快照重新查詢。刷新應保留 `selectedRoute`、當前排序字段與方向，不因刷新更新常用路線 `usage_count` 或 `last_used_at`。

原因：
- 下拉刷新語義是更新當前結果，不是選擇或重新使用一條常用路線。
- 若刷新也累計使用次數，常用路線排序會被等待 ETA 的重複刷新放大，偏離真實使用意圖。

### Decision: 僅收斂區塊間距，不壓縮核心控件尺寸

頂部標題列到 `常用路線` 區塊的間距建議落在 12dp 到 16dp；`查詢` 按鈕到底部第一個可見結果控件的有效間距也落在 12dp 到 16dp。結果區內部相鄰輔助控件間距可控制在 6dp 到 10dp。

不調整以下內容：
- 根容器 20dp 左右內距。
- `查詢`、`管理路線`、排序 chips 的最小觸控高度。
- 結果卡片內部路線、等候、價格、耗時、步行距離的可讀字號。

替代方案：
- 全局降低頁面 padding。這會讓界面貼邊，不符合風格指南。
- 壓縮結果卡片高度。這會降低掃讀舒適度，且可能破壞 5 到 6 條完整卡片的可讀目標。

### Decision: 管理入口使用主綠色，層級靠尺寸和位置區分

`管理路線` 按鈕改用與 `查詢` 主按鈕相同的主綠色背景與高對比文字。它仍位於標題列右側，使用較短文字、較緊湊寬度和普通標題列位置，避免與頁面中央的 `查詢` 主操作競爭。

替代方案：
- 保持琥珀色。這與用戶指定不符，也讓主頁操作色過於分散。
- 改成純文字入口。這會降低管理入口可發現性，與既有 `route-management-actions` 的按鈕感要求衝突。

### Decision: 臨時路線保存對話框使用帶外框的表單容器

將兩個臨時路線保存入口統一到同一套對話框 view 建立邏輯。對話框內容使用垂直容器承載：

```text
保存為常用
起點 -> 終點（輔助預覽，可單行省略）
[ 常用路線名稱                         ]
取消                                  保存
```

路線名稱輸入使用 `TextInputLayout` 的 outlined box 樣式或等效 Material 外框樣式，不直接把 bare underline 輸入框作為 dialog view。輸入框預填 `起點 -> 終點`，保留 select-all-on-focus；長文本以單行水平滾動或尾部省略處理，不與 hint、外框、底線或錯誤提示重疊。錯誤提示顯示在輸入框下方，並保留足夠 bottom padding，避免擠到 dialog actions。

替代方案：
- 只增加 `setPadding()`。這可能改善單個機型，但無法保證長名稱、不同字體縮放或錯誤提示下不重疊。
- 直接跳轉新增路線頁。這能使用既有完整表單，但保存臨時路線會變成跨頁流程，超出本次輕量 UI 修正範圍。

## Risks / Trade-offs

- [Risk] `SwipeRefreshLayout` 可能需要新增依賴。→ Mitigation: 只加入 AndroidX 官方最小依賴，並通過 Gradle build 驗證。
- [Risk] 刷新回調與正在進行的查詢交錯。→ Mitigation: 復用既有 `querySequence` 和 `cancelProgressiveQueries()`，刷新期間禁用再次刷新。
- [Risk] 間距收斂過度導致頁面擁擠。→ Mitigation: 只改區塊間距，保留控件高度與卡片內部可讀尺寸，並用截圖或模擬器目視驗證。
- [Risk] 管理入口與查詢按鈕同色後層級變近。→ Mitigation: 依靠標題列位置、尺寸和主操作所在區域區分層級，不把管理入口擴展成全寬主按鈕。
- [Risk] 保存對話框抽取共用 view 時兩個入口的保存後行為混在一起。→ Mitigation: 只共用輸入展示和校驗觸發結構，保存成功回調仍由各入口傳入並保持現有語義。
