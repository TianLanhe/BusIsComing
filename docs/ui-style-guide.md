# BusIsComming UI 风格指南

## 风格定位

BusIsComming 的默认界面风格是：安静实用的现代通勤工具。

这个风格优先服务每天反复使用的路线查询场景。界面应该清楚、轻快、可信赖，避免营销化、装饰化或后台表格化。后续通过 OpenSpec 新增或改造页面时，默认采用本指南，除非对应 change 明确说明偏离原因。

## 设计原则

- 信息优先：优先突出路线、候车时间、价格、总耗时和步行距离，不用装饰元素抢占注意力。
- 阅读舒适：移动端字号以可读为先，不为了一屏塞更多行而牺牲核心信息可读性。
- 层次清楚：主信息大、深、靠前；辅助信息小、浅、靠后；操作按钮按主次分级。
- 轻量现代：使用卡片、状态提示、图标按钮、淡入/滑入等轻量反馈提升现代感，但不做夸张动画。
- 通勤语境：视觉语言应接近日常出行工具，而不是营销落地页、后台管理系统或纯表格工具。
- 一致沉淀：新页面、新组件、新交互应复用本指南里的颜色、圆角、间距、动效和信息层级。

## 视觉语言

默认保留当前深青绿作为主色，琥珀色作为辅助操作色，并增加更清晰的中性色层次。

- 主色用于核心操作、关键状态和选中态。
- 辅助色用于次级但需要明显按钮感的操作，例如管理入口或强调型快捷入口。
- 背景使用浅色中性表面，不使用深色大面积背景。
- 卡片使用轻微描边或浅阴影表达层次，不使用厚重边框。
- 圆角保持克制，普通卡片建议 8dp 左右，主要操作按钮可略大，但不要全局过度圆润。
- 不使用渐变球、装饰光斑、无意义插画或大面积宣传型 hero。

## 主页面

主页面应该像一个高频使用的查询工具，第一屏直接提供路线选择、查询动作和结果状态。

推荐结构：

```text
┌────────────────────────────┐
│ 巴士查询          管理入口 │
├────────────────────────────┤
│ 常用路线选择区域            │
│ 路线名称                    │
│ 起点 -> 终点                │
├────────────────────────────┤
│ 查询按钮                    │
├────────────────────────────┤
│ 排序 / 查询状态 / 结果提示   │
├────────────────────────────┤
│ 路线结果卡片列表             │
└────────────────────────────┘
```

### 路线结果

查询结果默认使用路线结果卡片，不再优先使用五列表格。

结果卡片的信息优先级：

1. 路线号或路线组合
2. 候车时间
3. 价格
4. 总耗时
5. 步行距离

推荐卡片结构：

```text
┌────────────────────────────────────┐
│ 82X        约 4 分钟        HK$ 6.5 │
│            总耗时 32 分钟  步行 180米 │
└────────────────────────────────────┘
```

卡片字号建议：

- 路线号：20sp 到 24sp，粗体。
- 候车时间：18sp 到 20sp，核心状态。
- 价格：15sp 到 16sp。
- 总耗时、步行距离：13sp 到 14sp。

结果列表一屏优先显示 5 到 6 条完整结果。若与可读字号冲突，优先保证字体舒适。

### 排序与状态

- 排序入口可使用 chips、分段控件或紧凑文本按钮，不应依赖难以发现的表头点击。
- 查询中应有明确但不阻塞的反馈，例如按钮内进度、状态条或结果区 loading placeholder。
- 查询成功后结果卡片淡入或轻微上移进入。
- 查询失败和无结果应使用状态卡或空状态，不只显示一行普通文本。

## 路线编辑页

路线编辑页应该像一个轻量表单，而不是后台配置页。

- 页面标题明确区分新增、编辑、克隆。
- 路线名称、起点、终点输入框保持清晰的 helper text 或 hint。
- 起点和终点是一组关联字段，视觉上应比普通上下排列更有关系感。
- 交换起点和终点使用图标按钮，不使用整行文字按钮。

推荐交换控件结构：

```text
[ 起点输入框                         ]  ↕
[ 终点输入框                         ]
```

交换图标要求：

- 使用弯曲双向箭头语义。
- 放在两个输入框右侧中线附近，点击区域不小于 48dp。
- 提供无障碍说明，例如“交换起点和终点”。
- 点击后可使用 150ms 到 250ms 的旋转或淡入反馈。
- 交换后仍执行原有地点选择和保存校验。

地点搜索要求：

- 搜索加载反馈不阻塞输入。
- 新输入应取消旧搜索反馈。
- 候选列表、无结果、失败状态要清楚区分。

## 路线管理页

路线管理页与主页面保持同一风格，使用卡片列表表达路线配置。

推荐结构：

```text
路线管理                         新增
┌──────────────────────────────┐
│ 上学                           │
│ 漁灣邨 -> 北角衛理堂             │
│ 编辑   克隆   删除               │
└──────────────────────────────┘
```

- 新增路线应是页面内明确主操作，可使用顶部按钮或 FAB。
- 编辑、克隆、删除是卡片内次级操作。
- 删除必须保留二次确认。
- 空状态应提供新增入口，而不是只提示“暂无路线”。

## 动效与反馈

动效目标是让用户感受到状态变化，不是展示动画本身。

推荐动效：

- 页面进入：轻微 fade 或 slide。
- 查询中：按钮进度、状态条或结果区 placeholder。
- 结果出现：列表整体淡入，卡片轻微错峰进入。
- 排序切换：选中态变化和箭头方向变化。
- 起终点交换：图标轻微旋转，两个输入框内容快速切换。

动效约束：

- 单个反馈通常控制在 150ms 到 300ms。
- 不阻塞输入、滚动或页面返回。
- 不做循环装饰动画。
- 不因为动画导致布局跳动。

## Android 实现取向

当前项目默认使用 XML + AppCompat + Material Components。后续 UI 改造优先沿用现有技术栈，避免为了单个页面贸然引入 Compose。

可以考虑引入 Compose 的情况：

- 新页面需要复杂状态驱动布局，XML 实现会明显膨胀。
- 需要更复杂、可维护的转场或列表动画。
- 计划逐步把多个页面迁移到同一套 Compose UI 体系。

不建议引入 Compose 的情况：

- 只是调整字号、颜色、圆角、卡片和图标按钮。
- 只是重排现有表单或 RecyclerView 列表。
- 迁移会导致 XML 和 Compose 混用范围过大，却没有清晰收益。

若 OpenSpec change 决定引入 Compose，需要在 `design.md` 说明原因、迁移范围、与现有 XML 页面如何共存，以及验证方式。

## OpenSpec 使用约定

涉及页面或交互的 OpenSpec change 应在 `design.md` 中引用本指南，并说明：

- 当前页面使用的主风格是否为“安静实用的现代通勤工具”。
- 是否采用结果卡片、状态卡、图标按钮或轻量动效。
- 是否有偏离本指南的原因。
- 需要验证的视觉目标，例如一屏显示 5 到 6 条结果、核心字号是否舒适、触控区域是否不小于 48dp。

涉及 UI 的 `tasks.md` 应包含至少一项视觉或交互验证任务。Android 实现完成后，优先通过构建和真机或模拟器截图确认关键页面没有文字重叠、布局跳动或按钮不可辨识的问题。
