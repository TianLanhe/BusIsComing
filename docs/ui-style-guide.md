# BusIsComing UI 風格指南

## 風格定位

BusIsComing 的預設界面風格是：安靜實用的現代通勤工具。

這個風格優先服務每天反覆使用的路線查詢場景。界面應該清楚、輕快、可信賴，避免營銷化、裝飾化或後台表格化。後續通過 OpenSpec 新增或改造頁面時，預設採用本指南，除非對應 change 明確説明偏離原因。

## 設計原則

- 資訊優先：優先突出路線、候車時間、價格、總耗時和步行距離，不用裝飾元素搶佔注意力。
- 閲讀舒適：移動端字號以可讀為先，不為了一屏塞更多行而犧牲核心資訊可讀性。
- 層次清楚：主資訊大、深、靠前；輔助資訊小、淺、靠後；操作按鈕按主次分級。
- 輕量現代：使用卡片、狀態提示、圖示按鈕、淡入/滑入等輕量反饋提升現代感，但不做誇張動畫。
- 通勤語境：視覺語言應接近日常出行工具，而不是營銷落地頁、後台管理系統或純表格工具。
- 一致沉澱：新頁面、新組件、新互動應複用本指南里的顏色、圓角、間距、動效和資訊層級。

## 視覺語言

預設保留當前深青綠作為主色，琥珀色作為輔助操作色，並增加更清晰的中性色層次。

- 主色用於核心操作、關鍵狀態和選中態。
- 輔助色用於次級但需要明顯按鈕感的操作，例如管理入口或強調型快捷入口。
- 背景使用淺色中性表面，不使用深色大面積背景。
- 卡片使用輕微描邊或淺陰影表達層次，不使用厚重邊框。
- 圓角保持克制，普通卡片建議 8dp 左右，主要操作按鈕可略大，但不要全域過度圓潤。
- 不使用漸變球、裝飾光斑、無意義插畫或大面積宣傳型 hero。

## 主頁面

主頁面應該像一個高頻使用的查詢工具，第一屏直接提供路線選擇、查詢動作和結果狀態。

推薦結構：

```text
┌────────────────────────────┐
│ 巴士查詢          管理入口 │
├────────────────────────────┤
│ 常用路線選擇區域            │
│ 路線名稱                    │
│ 起點 -> 終點                │
├────────────────────────────┤
│ 查詢按鈕                    │
├────────────────────────────┤
│ 排序 / 查詢狀態 / 結果提示   │
├────────────────────────────┤
│ 路線結果卡片清單             │
└────────────────────────────┘
```

### 路線結果

查詢結果預設使用路線結果卡片，不再優先使用五清單格。

結果卡片的資訊優先級：

1. 路線號或路線組合
2. 候車時間
3. 價格
4. 總耗時
5. 步行距離

推薦卡片結構：

```text
┌────────────────────────────────────┐
│ 82X        約 4 分鐘        HK$ 6.5 │
│            總耗時 32 分鐘  步行 180米 │
└────────────────────────────────────┘
```

卡片字號建議：

- 路線號：20sp 到 24sp，粗體。
- 候車時間：18sp 到 20sp，核心狀態。
- 價格：15sp 到 16sp。
- 總耗時、步行距離：13sp 到 14sp。

結果清單一屏優先顯示 5 到 6 條完整結果。若與可讀字號衝突，優先保證字體舒適。

### 排序與狀態

- 排序入口可使用 chips、分段控件或緊湊文本按鈕，不應依賴難以發現的表頭點擊。
- 查詢中應有明確但不阻塞的反饋，例如按鈕內進度、狀態條或結果區 loading placeholder。
- 查詢成功後結果卡片淡入或輕微上移進入。
- 查詢失敗和無結果應使用狀態卡或空狀態，不只顯示一行普通文本。

## 路線編輯頁

路線編輯頁應該像一個輕量表單，而不是後台配置頁。

- 頁面標題明確區分新增、編輯、複製。
- 路線名稱、起點、終點輸入框保持清晰的 helper text 或 hint。
- 起點和終點是一組關聯字段，視覺上應比普通上下排列更有關係感。
- 交換起點和終點使用圖示按鈕，不使用整行文字按鈕。

推薦交換控件結構：

```text
[ 起點輸入框                         ]  ↕
[ 終點輸入框                         ]
```

交換圖示要求：

- 使用彎曲雙向箭頭語義。
- 放在兩個輸入框右側中線附近，點擊區域不小於 48dp。
- 提供無障礙説明，例如“交換起點和終點”。
- 點擊後可使用 150ms 到 250ms 的旋轉或淡入反饋。
- 交換後仍執行原有地點選擇和儲存校驗。

地點搜尋要求：

- 搜尋加載反饋不阻塞輸入。
- 新輸入應取消舊搜尋反饋。
- 候選清單、無結果、失敗狀態要清楚區分。

## 路線管理頁

路線管理頁與主頁面保持同一風格，使用卡片清單表達路線配置。

推薦結構：

```text
路線管理                         新增
┌──────────────────────────────┐
│ 上學                           │
│ 漁灣邨 -> 北角衞理堂             │
│ 編輯   複製   刪除               │
└──────────────────────────────┘
```

- 新增路線應是頁面內明確主操作，可使用頂部按鈕或 FAB。
- 編輯、複製、刪除是卡片內次級操作。
- 刪除必須保留二次確認。
- 空狀態應提供新增入口，而不是隻提示“暫無路線”。

## 動效與反饋

動效目標是讓用戶感受到狀態變化，不是展示動畫本身。

推薦動效：

- 頁面進入：輕微 fade 或 slide。
- 查詢中：按鈕進度、狀態條或結果區 placeholder。
- 結果出現：清單整體淡入，卡片輕微錯峯進入。
- 排序切換：選中態變化和箭頭方向變化。
- 起終點交換：圖示輕微旋轉，兩個輸入框內容快速切換。

動效約束：

- 單個反饋通常控制在 150ms 到 300ms。
- 不阻塞輸入、滾動或頁面返回。
- 不做循環裝飾動畫。
- 不因為動畫導致佈局跳動。

## Android 實現取向

當前專案預設使用 XML + AppCompat + Material Components。後續 UI 改造優先沿用現有技術棧，避免為了單個頁面貿然引入 Compose。

可以考慮引入 Compose 的情況：

- 新頁面需要複雜狀態驅動佈局，XML 實現會明顯膨脹。
- 需要更復雜、可維護的轉場或清單動畫。
- 計劃逐步把多個頁面遷移到同一套 Compose UI 體系。

不建議引入 Compose 的情況：

- 只是調整字號、顏色、圓角、卡片和圖示按鈕。
- 只是重排現有表單或 RecyclerView 清單。
- 遷移會導致 XML 和 Compose 混用範圍過大，卻沒有清晰收益。

若 OpenSpec change 決定引入 Compose，需要在 `design.md` 説明原因、遷移範圍、與現有 XML 頁面如何共存，以及驗證方式。

## OpenSpec 使用約定

涉及頁面或互動的 OpenSpec change 應在 `design.md` 中引用本指南，並説明：

- 當前頁面使用的主風格是否為“安靜實用的現代通勤工具”。
- 是否採用結果卡片、狀態卡、圖示按鈕或輕量動效。
- 是否有偏離本指南的原因。
- 需要驗證的視覺目標，例如一屏顯示 5 到 6 條結果、核心字號是否舒適、觸控區域是否不小於 48dp。

涉及 UI 的 `tasks.md` 應包含至少一項視覺或互動驗證任務。Android 實現完成後，優先通過構建和真機或模擬器截圖確認關鍵頁面沒有文字重疊、佈局跳動或按鈕不可辨識的問題。
