# BusIsComming 规格说明

## 1. 背景和目标

### 1.1 背景

BusIsComming 是一个面向巴士出行查询场景的 Android App。用户可以预先保存常用的起点和终点地址，并在主界面选择某条已保存路线后查询可用巴士路线、价格和预计车辆到站时间。

第一版为本地 MVP，重点验证核心交互和页面结构：

- 路线配置管理流程是否顺畅。
- 主界面是否能基于已保存路线完成查询。
- 查询结果表格是否能清晰展示路线、价格和预计等候时间。
- 排序交互是否符合使用预期。

真实巴士数据后续来自 HTTP API。当前阶段先使用本地 Mock 数据，且 Mock 数据需要根据起点和终点返回不同结果。

### 1.2 目标

第一版需要达成以下目标：

- 支持本地新增、查看、修改、删除路线配置。
- 支持在主界面从已保存路线配置中选择一个起终点组合。
- 支持点击查询后展示巴士路线结果。
- 支持按价格和预计等候时间排序。
- 支持无配置、无查询结果等基础空状态。
- 预留 HTTP API 接入位置，后续替换 Mock 数据源时尽量不影响 UI。

### 1.3 非目标

第一版不包含：

- 地图选点。
- 地址搜索和地址联想。
- 真实巴士 HTTP API。
- 登录、账号系统、云同步。
- 推送提醒。
- 多语言。
- 专项深色模式适配。
- 复杂换乘路线规划。

## 2. 用户场景

### 2.1 首次使用并新增路线

用户第一次打开 App，本地没有任何路线配置。主界面展示空状态，提示用户新增路线。用户点击新增入口，进入路线编辑页，填写路线名称、起点地址和终点地址，保存后可在主界面选择该路线。

### 2.2 查询已保存路线的巴士信息

用户打开 App 后，主界面展示已保存路线下拉列表。用户选择一条路线并点击查询，App 根据该路线的起点和终点加载巴士路线结果，并以表格形式展示路线、价格和预计等候时间。

### 2.3 按价格选择路线

用户查询到多条巴士路线后，点击表格中的“价格”表头。结果按港币价格升序排列。用户再次点击“价格”表头，结果按港币价格降序排列。

### 2.4 按等待时间选择路线

用户查询到多条巴士路线后，点击表格中的“预计等候时间”表头。结果按车辆到站等待分钟数升序排列。用户再次点击该表头，结果按等待时间降序排列。

### 2.5 修改常用路线

用户进入路线管理页，选择某条路线配置进行编辑。编辑页带入已有路线名称、起点地址和终点地址。用户修改后保存，返回管理页和主界面时均能看到更新后的配置。

### 2.6 删除不再使用的路线

用户进入路线管理页，点击某条路线的删除操作。App 弹出二次确认。用户确认后，该路线从本地删除；用户取消后，不产生任何变化。

### 2.7 查询无结果路线

用户选择一条起终点组合后点击查询。如果本地 Mock 数据没有匹配结果，结果区域展示“暂无可用巴士路线”。

## 3. 功能需求

### 3.1 路线配置管理

**FR-001 新增路线配置**

- 用户可以新增路线配置。
- 路线配置包含路线名称、起点地址、终点地址。
- 三个字段均必填。
- 保存成功后，路线配置写入本地存储。

**FR-002 查看路线配置列表**

- 用户可以进入路线管理页查看所有已保存路线配置。
- 每条配置展示路线名称，以及“起点地址 -> 终点地址”。

**FR-003 编辑路线配置**

- 用户可以编辑已有路线配置。
- 编辑页需要回填路线名称、起点地址、终点地址。
- 保存后更新本地存储。

**FR-004 删除路线配置**

- 用户可以删除已有路线配置。
- 删除前必须展示二次确认弹窗。
- 用户确认后删除；用户取消后保持原数据不变。

**FR-005 本地持久化**

- 路线配置只保存在本地设备。
- App 重启后，已保存路线配置仍可查看和选择。

### 3.2 主界面查询

**FR-006 主界面展示已保存路线**

- 主界面展示路线配置下拉列表。
- 下拉项展示格式为：

```text
路线名称：起点地址 -> 终点地址
```

**FR-007 无路线配置空状态**

- 当本地没有任何路线配置时，主界面展示空状态。
- 空状态需要提供新增路线入口。

**FR-008 查询巴士路线**

- 用户选择路线配置后，可以点击查询按钮。
- 查询参数来自所选路线配置的起点地址和终点地址。
- 第一版通过本地 Mock 数据返回查询结果。

**FR-009 查询结果展示**

- 查询结果以表格形式展示。
- 表格包含三列：
  - 路线
  - 价格
  - 预计等候时间

**FR-010 无查询结果空状态**

- 当查询结果为空时，展示“暂无可用巴士路线”。

### 3.3 排序

**FR-011 按价格排序**

- 用户点击“价格”表头后，结果按价格排序。
- 第一次点击为升序。
- 再次点击同一表头切换为降序。

**FR-012 按预计等候时间排序**

- 用户点击“预计等候时间”表头后，结果按等待分钟数排序。
- 第一次点击为升序。
- 再次点击同一表头切换为降序。

**FR-013 排序状态展示**

- 当前排序列需要展示排序方向。
- 建议展示为：
  - 价格 ↑
  - 价格 ↓
  - 预计等候时间 ↑
  - 预计等候时间 ↓

### 3.4 展示格式

**FR-014 价格展示**

- 价格单位固定为港币。
- 展示格式为 `HK$金额`。
- 示例：`HK$6.0`、`HK$12.5`。

**FR-015 预计等候时间展示**

- 预计等候时间表示车辆到站时间。
- 展示格式为 `约 N 分钟`。
- 示例：`约 3 分钟`、`约 12 分钟`。

### 3.5 权限

**FR-016 网络权限预留**

- 第一版需要在 Manifest 中加入 `android.permission.INTERNET`。
- 当前 Mock 查询不使用网络，但后续 HTTP API 接入会使用该权限。

## 4. 非功能需求

### 4.1 技术栈

- Android 原生应用。
- 语言使用 Kotlin。
- UI 使用 XML + AppCompat + Material Components。
- 本地存储使用 SQLiteOpenHelper。
- 列表和结果表格使用 RecyclerView。

### 4.2 可维护性

- 巴士查询逻辑必须通过 Repository 接口抽象。
- UI 不应直接依赖 Mock 数据实现。
- 后续替换真实 HTTP API 时，应优先新增 `HttpBusRouteRepository`，避免大范围修改 UI 代码。

### 4.3 可用性

- 页面需要清晰区分空状态、表单输入状态和查询结果状态。
- 表单字段为空时需要给出明确错误提示。
- 删除操作需要二次确认。
- UI 简洁，但不能只使用裸控件堆叠，应具备基本间距、层级和 Material 风格。

### 4.4 性能

- 第一版数据量较小，本地查询和排序应在用户点击后即时完成。
- 路线配置列表和查询结果使用 RecyclerView，以便未来支持更多数据。

### 4.5 兼容性

- 遵循当前工程配置的 minSdk 和 targetSdk。
- 不引入重量级依赖，避免增加构建和维护成本。

### 4.6 数据安全和隐私

- 路线配置仅存储在本地设备。
- 第一版不上传用户输入地址。
- 第一版不采集用户个人信息。

### 5.2 路线配置数据结构

```kotlin
data class RouteConfig(
    val id: Long,
    val name: String,
    val origin: String,
    val destination: String
)
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | Long | 本地数据库主键 |
| name | String | 用户自定义路线名称 |
| origin | String | 起点地址 |
| destination | String | 终点地址 |

### 5.3 巴士查询结果数据结构

```kotlin
data class BusRouteOption(
    val routeName: String,
    val priceHkd: Double,
    val waitMinutes: Int
)
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| routeName | String | 巴士路线名称 |
| priceHkd | Double | 港币价格 |
| waitMinutes | Int | 预计车辆到站等待分钟数 |

### 5.4 本地数据库设计

表名：

```sql
route_configs
```

字段：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
name TEXT NOT NULL
origin TEXT NOT NULL
destination TEXT NOT NULL
created_at INTEGER NOT NULL
updated_at INTEGER NOT NULL
```

约束：

- `name`、`origin`、`destination` 不允许为空字符串。
- `created_at` 和 `updated_at` 使用毫秒时间戳或秒级时间戳，具体实现保持一致即可。

### 5.5 路线配置仓库 API

```kotlin
class RouteConfigRepository {
    fun getAll(): List<RouteConfig>
    fun getById(id: Long): RouteConfig?
    fun insert(name: String, origin: String, destination: String): Long
    fun update(config: RouteConfig)
    fun delete(id: Long)
}
```

行为说明：

- `getAll()` 返回全部路线配置，建议按更新时间倒序或创建时间倒序。
- `getById(id)` 找不到时返回 `null`。
- `insert(...)` 返回新插入记录的 id。
- `update(config)` 只更新已存在记录。
- `delete(id)` 删除对应记录，记录不存在时不应崩溃。

### 5.6 巴士查询仓库 API

```kotlin
interface BusRouteRepository {
    fun searchRoutes(origin: String, destination: String): List<BusRouteOption>
}
```

第一版实现：

```kotlin
class MockBusRouteRepository : BusRouteRepository {
    override fun searchRoutes(origin: String, destination: String): List<BusRouteOption> {
        // 根据 origin + destination 返回本地写死数据
    }
}
```

后续真实 API 实现：

```kotlin
class HttpBusRouteRepository : BusRouteRepository {
    override fun searchRoutes(origin: String, destination: String): List<BusRouteOption> {
        // 调用真实 HTTP API，并映射为 BusRouteOption
    }
}
```

### 5.7 Mock 数据规则

第一版至少包含以下 Mock 匹配：

```text

```

返回：

| --- | --- | --- |
| 82 | 6.0 | 4 |
| 8X | 7.2 | 9 |
| 780 | 10.5 | 13 |

```text
兴华二村丰兴楼 -> 渔湾村渔进楼
```

返回：

| routeName | priceHkd | waitMinutes |
| --- | --- | --- |
| 82 | 6.0 | 6 |
| 8X | 7.2 | 11 |

其他未匹配的起点和终点返回空列表。

### 5.8 排序状态

```kotlin
enum class SortField {
    PRICE,
    WAIT_TIME
}

enum class SortDirection {
    ASC,
    DESC
}
```

排序规则：

- 初始状态不强制排序，按查询结果原始顺序展示。
- 点击新排序字段时，默认升序。
- 点击当前排序字段时，切换升序和降序。

### 5.9 未来 HTTP API 预留

后续接入真实 API 时需要补充：

- 请求 URL。
- HTTP 方法。
- 请求参数。
- 鉴权方式。
- 返回 JSON 结构。
- 错误码处理。
- 超时策略。
- 重试策略。
- 字段映射规则。

推荐技术：

- OkHttp 用于 HTTP 请求。
- kotlinx.serialization 或 org.json 用于 JSON 解析。
- Kotlin Coroutines 或后台线程用于避免主线程网络请求。

## 6. 边界情况

### 6.1 本地没有路线配置

系统表现：

- 主界面不展示无意义的空下拉。
- 展示空状态和新增路线入口。
- 查询按钮不可用或不展示。

### 6.2 表单字段为空

系统表现：

- 保存时校验路线名称、起点地址和终点地址。
- 任一字段为空时阻止保存。
- 在对应输入框展示错误提示。

### 6.3 输入内容只有空格

系统表现：

- 保存前对输入执行 trim。
- trim 后为空时按空字段处理。

### 6.4 删除路线后主界面仍选中该路线

系统表现：

- 主界面恢复时重新加载路线配置。
- 如果原选中路线已不存在，则清空选择或选中列表第一项。
- 如果删除后没有任何路线配置，则展示主界面空状态。

### 6.5 编辑时目标记录不存在

系统表现：

- 如果根据 id 找不到路线配置，应提示记录不存在。
- 不应崩溃。
- 可以返回管理页。

### 6.6 查询结果为空

系统表现：

- 隐藏或清空旧结果。
- 展示“暂无可用巴士路线”。

### 6.7 重复路线配置

第一版不强制禁止重复。

系统表现：

- 用户可以保存相同名称、相同起点和终点的配置。
- 后续如需限制，可补充唯一性规则。

### 6.8 价格相同或等待时间相同

系统表现：

- 排序时允许相同值。
- 相同值之间的顺序不作强约束。
- 不应出现数据丢失或重复。

### 6.9 特殊字符输入

系统表现：

- 路线名称、起点地址和终点地址允许中文、英文、数字和常见符号。
- 输入内容作为纯文本存储和展示。

### 6.10 App 重启

系统表现：

- 已保存路线配置仍存在。
- 主界面可以重新加载本地配置。

## 7. 验收标准

### 7.1 路线配置管理验收

- 可以新增一条包含名称、起点、终点的路线配置。
- 新增后，路线管理页能看到该路线。
- App 重启后，该路线仍存在。
- 可以编辑路线配置，保存后列表展示更新后的数据。
- 可以删除路线配置。
- 删除前出现确认弹窗。
- 取消删除时数据保持不变。
- 确认删除时数据从列表移除。

### 7.2 主界面验收

- 首次无配置时，主界面展示空状态和新增路线入口。
- 有配置时，主界面下拉列表展示已保存路线。
- 下拉项格式符合“路线名称：起点地址 -> 终点地址”。
- 用户选择路线后点击查询，可以展示查询结果。

### 7.3 查询结果验收

- 对 `渔湾村渔进楼 -> 兴华二村丰兴楼` 查询时，展示 Mock 路线 82、8X、780。
- 对 `兴华二村丰兴楼 -> 渔湾村渔进楼` 查询时，展示 Mock 路线 82、8X。
- 未匹配起终点时，展示“暂无可用巴士路线”。
- 查询结果表格包含“路线”“价格”“预计等候时间”三列。
- 价格展示为 `HK$金额`。
- 等待时间展示为 `约 N 分钟`。

### 7.4 排序验收

- 点击“价格”表头，结果按价格升序排列。
- 再次点击“价格”表头，结果按价格降序排列。
- 点击“预计等候时间”表头，结果按等待分钟数升序排列。
- 再次点击“预计等候时间”表头，结果按等待分钟数降序排列。
- 当前排序字段展示对应方向。

### 7.5 技术验收

- 项目可以通过 Gradle 构建。
- Manifest 包含 `android.permission.INTERNET`。
- 巴士查询逻辑通过 `BusRouteRepository` 抽象。
- Mock 数据实现不直接写死在 Activity 中。
- 路线配置数据通过本地数据库持久化。

## 8. 测试计划

### 8.1 单元测试

优先覆盖：

- `MockBusRouteRepository` 根据起终点返回正确数据。
- 未匹配起终点时返回空列表。
- 价格升序、价格降序排序逻辑。
- 等待时间升序、等待时间降序排序逻辑。
- 表单输入 trim 和空值判断逻辑。

### 8.2 本地数据库测试

覆盖：

- 插入路线配置。
- 查询全部路线配置。
- 根据 id 查询路线配置。
- 更新路线配置。
- 删除路线配置。
- 删除不存在 id 时不崩溃。

### 8.3 手动功能测试

测试路径：

1. 首次打开 App，确认主界面展示空状态。
2. 从空状态进入新增路线流程。
3. 不填写字段直接保存，确认表单错误提示。
4. 填写路线名称、起点地址、终点地址并保存。
5. 返回主界面，确认下拉列表显示新增路线。
6. 点击查询，确认展示 Mock 查询结果。
7. 点击价格表头，确认升序和降序切换。
8. 点击预计等候时间表头，确认升序和降序切换。
9. 进入路线管理页，编辑刚才新增的路线。
10. 删除路线，确认弹窗出现。
11. 取消删除，确认数据仍存在。
12. 再次删除并确认，确认数据移除。
13. 删除全部路线后返回主界面，确认空状态恢复。

### 8.4 边界测试

覆盖：

- 输入全空格并保存。
- 输入较长路线名称和较长地址。
- 输入中文、英文、数字和常见符号。
- 查询未匹配起终点。
- 查询结果价格相同。
- 查询结果等待时间相同。
- App 重启后检查本地数据是否保留。
